# 岗位匹配模块开发清单

> 项目：职点迷津（CareerPilot）
> 模块范围：JD 中心 + 匹配报告
> 生成日期：2026-03-15
> 目标：基于当前已完成的登录、个人资料、简历中心，跑通“选择简历 + 录入 JD + 生成匹配报告”的第一条真实业务闭环。

---

## 1. 这次开发要解决什么问题

当前项目已经具备：

- 用户登录注册
- 个人资料与求职偏好
- 简历上传、解析、结构化展示、人工校正

但还缺少真正体现产品价值的下一跳：

1. 用户没有地方录入目标岗位 JD。
2. 系统无法判断“这份简历适不适合这个岗位”。
3. 后续的简历优化、模拟面试、投递追踪都缺少一个明确的目标岗位上下文。

所以岗位匹配模块的目标很明确：

1. 让用户能保存多个目标 JD。
2. 让用户能从已有简历中选择一份，与某个 JD 做匹配。
3. 让系统输出可解释的匹配报告，而不是只给一个黑箱分数。

## 4. 数据表设计

## 4.1 `job_descriptions` 表

用途：保存用户录入的 JD 原文和结构化结果。

建议字段如下：

| 字段名 | 类型 | 说明 | 备注 |
| --- | --- | --- | --- |
| `id` | UUID | JD 主键 | 主键 |
| `user_id` | UUID | 所属用户 | FK -> `users.id` |
| `title` | VARCHAR(255) | 岗位标题 | 必填 |
| `company` | VARCHAR(255) | 公司名称 | 选填 |
| `job_city` | VARCHAR(120) | 岗位城市 | 选填 |
| `employment_type` | VARCHAR(50) | 用工类型 | 选填，如全职/实习 |
| `source_name` | VARCHAR(80) | 来源平台 | 选填，如 Boss / 猎聘 |
| `source_url` | TEXT | 原始链接 | 选填 |
| `jd_text` | TEXT | 原始 JD 文本 | 必填 |
| `parse_status` | VARCHAR(20) | 结构化状态 | `pending / success / failed` |
| `parse_error` | TEXT | 结构化失败原因 | 可空 |
| `structured_json` | JSON | 结构化结果 | 可空 |
| `created_at` | TIMESTAMP | 创建时间 | 复用当前 mixin 风格 |
| `updated_at` | TIMESTAMP | 更新时间 | 复用当前 mixin 风格 |
| `created_by` | UUID | 创建人 | 复用当前 mixin 风格 |
| `updated_by` | UUID | 更新人 | 复用当前 mixin 风格 |

建议索引：

- `idx_job_descriptions_user_id`
- `idx_job_descriptions_parse_status`
- `idx_job_descriptions_created_at`

建议的 `structured_json` 结构：

```json
{
  "basic": {
    "title": "高级数据分析师",
    "company": "某互联网公司",
    "job_city": "上海",
    "employment_type": "全职"
  },
  "requirements": {
    "required_skills": ["Python", "SQL", "A/B Testing"],
    "preferred_skills": ["Tableau", "Airflow"],
    "required_keywords": ["用户增长", "数据建模", "指标体系"],
    "education": "本科及以上",
    "experience_min_years": 3
  },
  "responsibilities": [
    "负责业务指标体系搭建",
    "支持实验分析和复盘",
    "与产品和运营协作推进增长项目"
  ],
  "benefits": ["双休", "弹性办公"],
  "raw_summary": "偏增长分析方向，强调 Python、SQL 和业务协同能力"
}
```

说明：

- 第一版不要追求结构特别复杂，重点是能稳定抽出“技能、关键词、经验、学历、城市”。
- 后续接 AI 时也尽量沿用这套结构，避免前后端反复改字段。

## 4.2 `match_reports` 表

用途：保存某一份简历和某一个 JD 的匹配结果。

建议字段如下：

| 字段名 | 类型 | 说明 | 备注 |
| --- | --- | --- | --- |
| `id` | UUID | 报告主键 | 主键 |
| `user_id` | UUID | 所属用户 | FK -> `users.id` |
| `resume_id` | UUID | 参与匹配的简历 | FK -> `resumes.id` |
| `jd_id` | UUID | 目标 JD | FK -> `job_descriptions.id` |
| `status` | VARCHAR(20) | 报告生成状态 | `pending / success / failed` |
| `overall_score` | NUMERIC(5,2) | 综合分 | 0-100 |
| `rule_score` | NUMERIC(5,2) | 规则分 | 0-100 |
| `model_score` | NUMERIC(5,2) | 模型修正分 | 第一版固定为 0 |
| `dimension_scores_json` | JSON | 各维度分数 | 必填 |
| `gap_json` | JSON | 优势、短板、建议 | 必填 |
| `evidence_json` | JSON | 匹配证据 | 必填 |
| `error_message` | TEXT | 失败原因 | 可空 |
| `created_at` | TIMESTAMP | 创建时间 | |
| `updated_at` | TIMESTAMP | 更新时间 | |
| `created_by` | UUID | 创建人 | |
| `updated_by` | UUID | 更新人 | |

建议索引：

- `idx_match_reports_user_id`
- `idx_match_reports_jd_id`
- `idx_match_reports_resume_id`
- `idx_match_reports_status`
- `idx_match_reports_resume_jd`

建议的 `dimension_scores_json` 结构：

```json
{
  "required_skills": 32,
  "preferred_skills": 10,
  "responsibility_keywords": 18,
  "experience_level": 12,
  "education_and_location": 8
}
```

建议的 `gap_json` 结构：

```json
{
  "strengths": [
    {
      "label": "Python",
      "reason": "简历技能与项目经历均出现 Python",
      "severity": "high"
    }
  ],
  "gaps": [
    {
      "label": "A/B Testing",
      "reason": "JD 明确要求，但简历中未找到直接证据",
      "severity": "high"
    }
  ],
  "actions": [
    {
      "priority": 1,
      "title": "补充实验分析经历",
      "description": "在项目经历中补一段实验设计、指标监控或复盘案例"
    }
  ]
}
```

建议的 `evidence_json` 结构：

```json
{
  "matched_resume_fields": {
    "skills": ["Python", "SQL"],
    "projects": ["增长实验分析平台"],
    "work_experience": ["负责指标体系建设"]
  },
  "matched_jd_fields": {
    "required_skills": ["Python", "SQL"],
    "required_keywords": ["指标体系"]
  },
  "missing_items": ["A/B Testing", "Tableau"],
  "notes": ["简历中出现了分析经历，但实验设计证据不足"]
}
```

## 5. 后端接口设计

接口命名要尽量贴合当前仓库风格，优先走 REST 风格，响应继续复用当前统一响应格式。

## 5.1 JD 中心接口

### 1. 创建 JD

- 方法：`POST /jobs`
- 作用：保存 JD，并触发第一版结构化

请求体建议：

```json
{
  "title": "高级数据分析师",
  "company": "某互联网公司",
  "job_city": "上海",
  "employment_type": "全职",
  "source_name": "Boss直聘",
  "source_url": "https://example.com/job/123",
  "jd_text": "这里是完整 JD 文本"
}
```

### 2. 获取 JD 列表

- 方法：`GET /jobs`
- 作用：用于左侧列表展示
- 建议支持查询参数：`keyword`、`parse_status`

### 3. 获取 JD 详情

- 方法：`GET /jobs/{job_id}`
- 作用：获取原文、结构化结果、基础信息

### 4. 更新 JD

- 方法：`PUT /jobs/{job_id}`
- 作用：编辑标题、公司、JD 文本
- 约定：如果 `jd_text` 变化，自动重置 `parse_status` 并重新解析

### 5. 删除 JD

- 方法：`DELETE /jobs/{job_id}`
- 作用：删除 JD
- 约定：若已有匹配报告，先允许删除，再由报告页显示“关联 JD 已删除”不是一个好体验；更稳妥的做法是第一版限制“有报告的 JD 不允许删除”。

推荐第一版：

- 若 JD 已有关联报告，返回 `409 CONFLICT`
- 前端提示“请先删除相关报告或保留该 JD”

### 6. 重试 JD 结构化

- 方法：`POST /jobs/{job_id}/parse`
- 作用：当结构化失败后允许重试

## 5.2 匹配报告接口

### 7. 创建匹配报告

- 方法：`POST /jobs/{job_id}/match-reports`
- 作用：选择某份简历，为当前 JD 生成一份匹配报告

请求体建议：

```json
{
  "resume_id": "uuid"
}
```

### 8. 获取某个 JD 下的匹配报告列表

- 方法：`GET /jobs/{job_id}/match-reports`
- 作用：查看同一个 JD 的历史报告

### 9. 获取匹配报告详情

- 方法：`GET /match-reports/{report_id}`
- 作用：报告详情展示

### 10. 删除匹配报告

- 方法：`DELETE /match-reports/{report_id}`
- 作用：清理历史报告

## 8. 第一版评分规则怎么定

第一版一定要坚持两个原则：

1. 分数必须能解释。
2. 先规则化，再考虑 AI 修正。

## 8.1 总分构成

第一版总分满分 100，建议拆成 5 个维度：

| 维度 | 权重 | 说明 |
| --- | --- | --- |
| 必备技能覆盖 | 40 | JD 必备技能命中情况 |
| 加分技能覆盖 | 15 | JD 加分项命中情况 |
| 岗位职责 / 关键词命中 | 20 | JD 中职责和关键词在简历中的命中情况 |
| 经验级别匹配 | 15 | 简历经历是否接近 JD 的经验要求 |
| 学历 / 城市 / 语言等基础条件 | 10 | 只做轻量匹配，不做复杂硬过滤 |

公式：

```text
overall_score = required_skill_score
              + preferred_skill_score
              + responsibility_score
              + experience_score
              + base_condition_score
```

第一版：

- `rule_score = overall_score`
- `model_score = 0`

## 8.2 字段来源

### 简历侧数据来源

使用当前已存在的 `resumes.structured_json`，重点读取：

- `basic_info.summary`
- `work_experience`
- `projects`
- `skills.technical`
- `skills.tools`
- `skills.languages`
- `certifications`
- `education`

### JD 侧数据来源

使用 `job_descriptions.structured_json`，重点读取：

- `requirements.required_skills`
- `requirements.preferred_skills`
- `requirements.required_keywords`
- `requirements.education`
- `requirements.experience_min_years`
- `basic.job_city`
- `responsibilities`

## 8.3 具体打分规则

### A. 必备技能覆盖（40 分）

规则：

```text
命中数 / 必备技能总数 * 40
```

示例：

- JD 必备技能 4 个
- 简历命中 3 个
- 得分 = 3 / 4 * 40 = 30

命中策略：

- 先做大小写归一化
- 去除空格和常见符号差异
- 第一版先用同义词表兜底少量别名，如 `js -> javascript`

### B. 加分技能覆盖（15 分）

规则：

```text
命中数 / 加分技能总数 * 15
```

若 JD 没有加分技能：

- 该维度直接记满分 15

原因：

- 避免因为 JD 写得粗糙，让用户白白丢分。

### C. 岗位职责 / 关键词命中（20 分）

规则：

- 从 JD 的 `required_keywords + responsibilities` 提取关键词
- 在简历摘要、经历、项目中做文本命中统计
- 命中比例映射到 0-20 分

建议实现：

```text
命中关键词数 / 关键词总数 * 20
```

说明：

- 第一版不做复杂 NLP，关键词命中足够支撑 MVP。
- 若关键词明显重复，需要先去重再计分。

### D. 经验级别匹配（15 分）

规则建议：

- 简历文本中若能识别出“3年 / 5年 / 2年以上”这类信息，则与 JD 的 `experience_min_years` 比较
- 若简历经验年限 >= JD 要求，记满分
- 若略低于要求，按比例扣分
- 若无法识别，给 7 分默认中间分，并在报告中提示“经验信息证据不足”

推荐公式：

```text
if 无法识别:
  7
elif 简历年限 >= JD要求:
  15
else:
  max(0, 简历年限 / JD要求 * 15)
```

### E. 学历 / 城市 / 语言等基础条件（10 分）

建议拆成：

- 学历匹配：4 分
- 城市匹配：3 分
- 语言 / 证书 / 其他轻条件：3 分

规则：

- 命中则给分
- 未提及则不给满分，但也不要全扣死

推荐策略：

- 明确命中：给满
- 明确不符：给 0
- 简历未体现：给一半，并在短板项里提示“证据不足”

## 8.4 报告文案规则

报告里至少输出 4 块内容：

1. 总体结论
2. 优势项
3. 短板项
4. 下一步动作

文案先用模板生成即可：

### 总体结论模板

```text
这份简历与目标岗位的匹配度为 {score} 分，整体属于 {high/good/medium/low} 匹配。
当前优势集中在 {top strengths}，主要短板集中在 {top gaps}。
```

### 匹配档位建议

- `85-100`：高匹配
- `70-84`：较匹配
- `50-69`：中等匹配
- `0-49`：低匹配

## 11. 建议的最小可交付版本

如果想先快速落地，建议把第一版压缩到下面这个最小集：

1. 只支持手动粘贴 JD
2. 只支持从已有简历里选 1 份进行匹配
3. 分数完全由规则引擎给出
4. 报告解释先用模板生成
5. 只做一个页面 `/dashboard/jobs`

这样做的好处：

1. 开发路径最短
2. 结果最可控
3. 后续加 AI 时不需要推翻前面结构

## 12. 本模块完成后的自然下一步

岗位匹配模块完成后，最自然的下一步不是马上做面试，而是：

1. 基于 `match_reports` 生成 `optimization_reports`
2. 在匹配报告页增加“生成简历优化建议”入口
3. 再让投递追踪消费“目标 JD + 匹配状态”
4. 最后让模拟面试消费“简历 + JD + 匹配短板”

也就是说，这个模块一旦落地，后面三个模块都会更顺。

## 13. 最终结论

这次岗位匹配模块的核心不是“做一个新页面”，而是建立一个新的业务中枢：

- JD 是目标
- 简历是输入
- 匹配报告是判断层
- 优化建议、投递追踪、模拟面试都会从这里继续长出来

所以开发时一定坚持这两个原则：

1. 先把规则化、可解释、可保存的报告做出来。
2. 先保证闭环，再逐步引入 AI 增强。

只要按这份清单执行，岗位匹配模块就会是《职点迷津》真正进入“产品能力阶段”的第一个核心模块。
