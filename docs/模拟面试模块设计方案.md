# 《模拟面试模块》DEV_DOCUMENT 设计方案 V1（含开源项目借鉴）

## Summary

- 模拟面试模块不是独立聊天工具，而是 CareerPilot 四阶段闭环中的第四环：
  - `简历解析 -> 岗位匹配 -> 简历优化 -> 模拟面试 -> 面试复盘 -> 反哺简历优化`
- 本模块必须深度复用现有链路产物，尤其是 [match_report.py](/Users/zhengwenze/Desktop/codex/career-pilot/apps/api/app/services/match_report.py) 已生成的 `interview_blueprint_json`，避免重新做一套与岗位匹配脱节的“通用问答机器人”。
- 当前仓库已具备三项关键前置条件：
  - 统一 AI 调用入口：[ai_client.py](/Users/zhengwenze/Desktop/codex/career-pilot/apps/api/app/services/ai_client.py)
  - 已产出的面试蓝图字段：[match_ai.py](/Users/zhengwenze/Desktop/codex/career-pilot/apps/api/app/services/match_ai.py)
  - 前端页面与导航占位：[page.tsx](</Users/zhengwenze/Desktop/codex/career-pilot/apps/web/src/app/(dashboard)/dashboard/interviews/page.tsx>)、[nav-config.ts](/Users/zhengwenze/Desktop/codex/career-pilot/apps/web/src/config/nav-config.ts)
- 本设计只落地文本版模拟面试，不做语音输入、TTS、视频、情绪识别。

## Product Positioning

- 模块目标不是“生成一些面试题”，而是帮助用户验证三件事：
  - 简历里写出来的内容，能否在口头表达中站得住
  - 岗位匹配阶段识别出的核心差距，是否已经补强
  - 简历优化阶段写出的新内容，是否具备真实经历支撑
- 模块对上游的要求：
  - 必须绑定一份 `match_report`
  - 默认绑定一份已解析成功的 `resume`
  - 默认绑定一份已结构化成功的 `job_description`
- 模块对下游的产出：
  - 结构化面试复盘 `review_json`
  - 可直接回流到简历优化模块的 `follow_up_tasks`
  - 可回流到岗位模块的 `readiness updates`

## External Benchmarks

- [modamaan/Ai-mock-Interview](https://github.com/modamaan/Ai-mock-Interview)
  - 可借鉴点：按岗位角色与经验级别定制问题，保留最近面试记录，结束后输出详细反馈。
- [MockMate](https://github.com/shubham220420/MockMate)
  - 可借鉴点：强调基于角色、简历、经历做自然对话，而不是固定题库。
- [PrepWise](https://github.com/YathishGP003/PrepWise)
  - 可借鉴点：保留 transcript，按固定维度评分并生成反馈页。
- [AI-mock-interview](https://github.com/SHARUN266/AI-mock-interview)
  - 可借鉴点：保留 session history，支持个性化面试与复盘。
- [open-interview](https://github.com/dsdanielpark/open-interview)
  - 可借鉴点：明确把 resume 和 JD 作为题目生成输入，并提醒过多迭代会导致重复题与 token 浪费。
- 结论：
  - 不做开放式 Agent 漫游
  - 不做无限轮追问
  - 不做脱离简历/JD/匹配报告的通用聊天
  - 要做的是“岗位上下文化的有限状态多轮训练”

## Goals

- 输入真实求职上下文：
  - 用户简历
  - 用户目标岗位
  - 岗位匹配报告
  - 可选的简历优化会话结果
- 进行一场可控、可终止、可复盘的多轮文本面试
- 输出结构化复盘，并能反哺岗位匹配与简历优化模块

## Non-Goals

- 不做语音输入与语音评分
- 不做公开视频面试模拟
- 不做开放式知识问答陪练
- 不做题库 CMS 或社区内容平台
- 不在本期扩展新的前台公开复杂状态机

## User Stories

1. 用户在岗位匹配页看完 `match_report` 后，直接点击“开始模拟面试”，系统自动携带 `job_id + report_id + resume_id` 进入训练。
2. 用户完成 6 到 8 个主问题后，得到一份结构化面试复盘，明确知道哪些回答站得住、哪些地方还空。
3. 用户能看到自己之前练过哪些岗位、每次练习得分如何变化、哪些短板反复出现。
4. 用户可以把复盘中的“补证据建议”直接转成简历优化任务，而不是手动摘抄。

## End-to-End Workflow

1. 用户在岗位工作台选择一个 `match_report`
2. 前端请求创建 `mock_interview_session`
3. 后端校验：
   - `resume.parse_status == success`
   - `job.parse_status == success`
   - `match_report.status == success`
   - `match_report.stale_status == fresh`
4. 后端生成 `session plan`
   - 优先读取 `match_report.interview_blueprint_json`
   - 叠加 `gap_json`、`tailoring_plan_json`、可选 `resume_optimization_session`
5. 系统返回第一题和 session 基本信息
6. 用户提交文本回答
7. 后端调用 AI 编排器，只允许三种决策：
   - `follow_up`
   - `next_question`
   - `finish_and_review`
8. 达到结束条件后，session 进入 `reviewing`
9. 后端基于 transcript 和逐轮评估生成 `review_json`
10. 前端展示复盘，并提供两个 CTA：

- `去简历优化`
- `重新练一轮`

## Domain Model

### Core Aggregate

- `MockInterviewSession`
  - 一场绑定真实岗位与简历上下文的面试训练会话
- `MockInterviewTurn`
  - 一轮具体问答，包含题目、回答、评估和下一步决策

### Upstream Dependencies

- `Resume`
- `JobDescription`
- `MatchReport`
- `ResumeOptimizationSession` 可选

### Downstream Assets

- `review_json`
- `follow_up_tasks_json`
- `job_readiness_event`

## State Machine

### Session Status

- `active`
  - 会话已创建，正在提问或等待回答
- `reviewing`
  - 主问题已结束，正在生成复盘
- `completed`
  - 复盘已生成
- `failed`
  - 本场面试无法继续

### Turn Status

- `asked`
  - 题目已生成，等待用户回答
- `answered`
  - 用户已提交回答
- `evaluated`
  - 已完成本轮判定

### Allowed Transitions

- `session: active -> reviewing -> completed`
- `session: active -> failed`
- `session: reviewing -> failed`
- `turn: asked -> answered -> evaluated`

### Must-Stop Conditions

- `match_report` 已过期
- `resume` 或 `job` 已产生新版本
- AI 返回非法 JSON 且修复失败
- 当前 session 已完成或失败，但前端仍尝试继续提交回答

## Interview Planning Strategy

### Source Priority

1. `match_report.interview_blueprint_json.question_pack`
2. `match_report.interview_blueprint_json.focus_areas`
3. `match_report.gap_taxonomy_json.must_fix`
4. `match_report.tailoring_plan_json.must_add_evidence`
5. `resume_optimization_session.selected_tasks` 可选

### Question Composition

- 每场固定 `6-8` 个主问题
- 每题最多 `1` 次追问
- 主问题按以下比例抽取：
  - `40%` 核心短板验证
  - `30%` 优势展开
  - `20%` 项目深挖
  - `10%` 求职动机与岗位匹配

### Modes

- `general`
  - 综合模拟，默认模式
- `behavioral`
  - 行为面试
- `project_deep_dive`
  - 项目/经历深挖
- `technical`
  - 技术重点验证
- `hr_fit`
  - 动机、协作、稳定性、文化适配

### End Conditions

- 主问题用尽
- 达到 `max_turn_count`
- 连续两轮回答严重偏题且无有效信息
- 用户主动结束

## Scoring Model

### Fixed Dimensions

- `relevance_to_job`
  - 是否围绕目标岗位核心要求
- `specificity_of_evidence`
  - 是否给出真实场景、任务和动作
- `ownership_and_depth`
  - 是否说明自己承担了什么，而不是只说团队做了什么
- `structure_and_clarity`
  - 是否有清晰组织，是否可听懂
- `results_and_metrics`
  - 是否说明结果、影响、指标
- `risk_flags`
  - 是否出现夸大、空泛、避重就轻、与简历不一致

### Turn-Level Decision Rules

- 只说职责，不说结果：
  - 追问指标、结果、影响
- 只说技能，不说场景：
  - 追问具体项目和上下文
- 只说团队成果，不说个人贡献：
  - 追问个人 ownership
- 回答偏题：
  - 给一次纠偏追问，之后进入下一题
- 同题追问次数达到上限：
  - 强制进入下一题

- 回写岗位 readiness event

## AI Architecture

### Why Not Single Free-Form Agent

- 单一自由对话模型不可控
- 容易重复提问、跑题、忘记评分边界
- 很难稳定回写数据库结构

### Three AI Roles

1. `Interview Planner`
   - 负责把现有 `interview_blueprint` 扩成一场有限问题计划
2. `Turn Orchestrator`
   - 负责评估一轮回答并决定“追问 / 下一题 / 结束”
3. `Review Generator`
   - 负责全场复盘

### Shared Guardrails

- 严格 JSON 输出
- 不能编造简历里没有的经历、时间、指标
- 不能生成 schema 外字段
- 不能绕过后端状态机直接推进会话
- 每次回答只允许返回一个明确下一步

### Prompt 1: Interview Planner

系统指令目标：

- 将 `match_report` 中已有蓝图扩展为本场面试计划
- 不能脱离现有岗位上下文自由发散

输入 payload：

```json
{
  "resume_snapshot": {},
  "job_snapshot": {},
  "match_report_snapshot": {
    "fit_band": "partial",
    "gap_json": {},
    "tailoring_plan_json": {},
    "interview_blueprint_json": {}
  },
  "optimization_snapshot": {},
  "session_mode": "general",
  "constraints": {
    "max_questions": 6,
    "max_follow_ups_per_question": 1
  }
```

## Prompt Validation Rules

- Planner 失败：
  - session 创建失败，直接报错，不进入面试
- Turn Orchestrator 失败：
  - 当前回答提交失败，不自动跳题，不生成脏数据
- Review Generator 失败：
  - session 停留在 `reviewing` 后转 `failed`，提示用户稍后重试复盘

### JSON Repair Strategy

- 复用现有 [ai_client.py](/Users/zhengwenze/Desktop/codex/career-pilot/apps/api/app/services/ai_client.py) 的 strict JSON 解析逻辑
- 对必填字段缺失场景，允许一次 repair prompt 重试
- 超过一次仍失败，任务明确失败，不做自由文本兜底落库
