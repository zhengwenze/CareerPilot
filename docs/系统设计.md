## Problem Statement
Career Pilot《职点迷津》要把大模型能力封装成一个普通求职者能直接上手的应用，而不是一个“会聊天的 AI 工具箱”。产品只做四件事：解析简历、分析岗位匹配度、生成更贴近目标岗位的简历版本、以及围绕岗位和简历做一场真实可用的模拟面试。核心问题不是能不能调大模型，而是如何把四个模块串成一条可信、稳定、可解释、可迭代的求职工作流。

## What Makes This Cool
- 用户只上传一份简历和一份 JD，就能拿到“诊断报告 + 优化草案 + 面试训练”这一整条链路，而不是四个孤立工具。
- 大模型不直接替代规则，而是只在最适合的地方做校正、归因、改写和追问，既保留可控性，也能做出明显优于纯规则系统的体验。
- 每个模块都把中间产物结构化落库：解析结果、匹配证据、优化任务、面试蓝图都能互相复用，不会每一步都从零开始重新问模型。

## Constraints
- 单人开发，设备为 MacBook Air M4，本地资源有限，不能依赖重型训练或复杂自建向量/排序平台。
- 当前仓库后端已存在 `resume`、`match_report`、`resume_optimization_session`、`mock_interview_session/turn` 等对象和服务入口，设计应复用这些边界，不推翻当前对象模型。
- 系统必须防止大模型在“简历优化”和“面试反馈”阶段编造经历、项目、指标。
- MVP 必须优先做出闭环，而不是追求所有格式、所有岗位、所有面试形态一次到位。

## Premises
1. 这个产品的真正交付物不是“AI 对话”，而是“围绕目标岗位的一套结构化求职结果物”：解析 JSON、匹配报告、优化后的简历版本、结构化面试报告。
2. 四个模块必须共用同一份候选人事实底座，否则后续优化和面试一定会产生幻觉和前后矛盾。
3. 大模型在本产品中应该承担“校正、理解、重写、评审”角色，而不是承担“原始事实抽取器”角色。
4. 匹配分数必须可拆解、可解释、可追溯到证据，否则用户不信，也无法驱动后面的优化和面试。

## Approaches Considered
### Approach A: 规则优先最小闭环
Summary: 简历解析、岗位解析、匹配打分全部以规则和关键词覆盖为主，LLM 只用于文案总结和简历改写。
Effort: M
Risk: Low
Pros:
- 开发快，单人可控，容易调试。
- 输出稳定，成本低。
- 与现有 `resume_parser.py` 和 `match_engine.py` 非常贴合。
Cons:
- 对复杂版式、模糊表达、跨行语义理解能力弱。
- 岗位匹配容易流于关键词比对。
- 面试体验会偏模板化。
Reuses:
- `app/services/resume_parser.py`
- `app/services/match_engine.py`
- `match_report` 的结构化 JSON 字段

### Approach B: LLM First 智能代理流
Summary: 上传后直接让多模态/长上下文模型抽取简历和 JD，匹配、优化、面试全部主要依赖大模型评估。
Effort: M
Risk: High
Pros:
- 初期体验上限高，遇到非标准内容时更灵活。
- 开发初期写规则更少。
Cons:
- 幻觉风险高，尤其在优化简历时容易编造。
- 分数不可解释，难以稳定复现。
- 成本更高，回归测试更难。
Reuses:
- `resume_ai.py`
- `match_ai.py`
- `mock_interview_ai.py`

### Approach C: 规则骨架 + LLM 校正/评审的混合架构
Summary: 先用规则和轻量 NLP 建事实骨架，再用 LLM 做结构校正、语义补全、重排评审、表达改写、面试追问和总结。
Effort: L
Risk: Medium
Pros:
- 可控性、可解释性和智能体验平衡最好。
- 每一步都能做 fallback，不会因为某次模型异常让整条链路失效。
- 最适合逐步演进到更强能力。
Cons:
- 设计复杂度高于纯规则方案。
- 需要清楚划分哪些字段能改、哪些字段不能改。
Reuses:
- `resume_parser.py` 作为规则主链
- `resume_ai.py` 作为校正层
- `match_engine.py` + `match_ai.py` 做双层评分
- `resume_optimizer.py`、`mock_interview_ai.py` 做下游复用

## Recommended Approach
选择 Approach C。Career Pilot 的核心卖点不是“像 ChatGPT 一样能回答”，而是“像真实求职软件一样能给出可信结果”。这要求底层先有稳定事实层，再让 LLM 在事实边界内增强表达和评估。

## Detailed Design
### 1. 产品目标
- 一个输入：候选人上传简历并选择目标岗位。
- 四个输出：
  1. 标准化候选人画像
  2. 可解释的岗位匹配报告
  3. 面向目标岗位的定制简历版本
  4. 基于简历和岗位差距的模拟面试训练报告
- 一个核心原则：所有模块都围绕“候选人事实底座”运行。事实只能来自原简历原文、用户补充信息、或用户在面试中主动给出的内容。

### 2. 模块设计
#### 2.1 简历解析
输入：
- 文件：PDF / DOCX / 图片
- 用户语言偏好（可选）

处理链路：
1. 文件识别与路由
   - PDF 文本型：直接走文本抽取。
   - PDF 扫描型 / 图片：走 OCR。
   - DOCX：提取段落、表格、样式层级。
2. 版面标准化
   - 把页、块、行、表格统一转为 `document_blocks`。
   - 保留页码、bbox、块顺序、字体大小这类结构信息，供后续判断标题/列表/时间范围。
3. 文本清洗
   - 全角半角归一化、换行规范化、邮箱电话正则抽取、日期跨度识别、项目符号归一化。
4. 规则解析
   - 先切分 section：基本信息、教育、经历、项目、技能、证书。
   - 再做字段级抽取：学校、专业、学位、公司、岗位、时间段、项目名、技术栈、职责、成果。
5. LLM 校正
   - 输入原文片段 + 规则结果 + schema。
   - 只允许做：纠正 section 归属、合并重复项、补全明显漏掉的字段、规范化表达。
   - 禁止做：新增经历、杜撰技能、猜测时间。
6. 结构校验
   - Pydantic 校验。
   - 时间格式、联系方式、必填 section 数量校验。
7. 入库
   - `raw_text`
   - `structured_json`
   - `parse_artifacts_json`（建议新增，保存 OCR/块结构/置信度）
   - `parse_status`

传统方法适合：
- 文件类型识别
- OCR 路由
- 正则抽邮箱/电话/日期
- section heading 识别
- 技能词典匹配
- 校验和去重

大模型适合：
- 非标准版式下的 section 重归类
- 合并跨行语义片段
- 模糊字段补位
- 中英文混合简历的语义理解

建议输出 JSON：

#### 2.2 岗位匹配
输入：
- `resume.structured_json`
- `resume.raw_text`
- `job_description.structured_json`
- 可选：用户目标偏好（城市、薪资、岗位方向）

处理链路：
1. JD 结构化
   - 提取岗位名、城市、学历要求、年限要求、必须技能、加分项、职责、领域关键词、产出预期。
2. 候选人证据建模
   - 从简历生成 `candidate_evidence_profile`：
     - 标准技能表
     - 经历关键词
     - 项目关键词
     - 行业/场景标签
     - 年限估计
     - 教育层级
3. 规则打分
   - 必备项命中、年限覆盖、学历覆盖、地点覆盖、职责关键词覆盖。
4. 向量召回
   - 把 JD 的职责/要求切成多个 query。
   - 用 embedding 去简历经历 bullet / 项目 bullet 中找最相关证据片段。
5. 重排
   - 用 cross-encoder 或轻量 reranker 对召回证据排序，得到“JD 要求 <-> 简历片段”的最佳映射。
6. LLM 评审
   - 输入 top-K 证据对，输出：
     - 语义匹配解释
     - 是否存在“有经验但没写清”
     - 是否存在“JD 要求完全无证据”
     - 优先优化建议
7. 生成报告
   - `scorecard_json`
   - `gap_json`
   - `evidence_map_json`
   - `tailoring_plan_json`
   - `interview_blueprint_json`

推荐打分维度：
- 必备技能覆盖：30
- 核心职责相关性：20
- 项目/业务场景相关性：15
- 年限与职级匹配：10
- 学历与专业匹配：5
- 行业/domain 匹配：10
- 表达清晰度与可证据化程度：10

推荐评分框架：
```text
overall_score
= 0.65 * rule_score
+ 0.25 * semantic_score
+ 0.10 * llm_judge_score

rule_score
= sum(weight_i * normalized_dimension_i)

semantic_score
= average(top_k_evidence_similarity) * 100

llm_judge_score
= grounded rubric score on:
  - evidence sufficiency
  - responsibility fit
  - domain fit
  - credibility / non-hallucination
```

更具体的可落地公式：
```text
score_required_skill = matched_required_skills / total_required_skills * 100
score_responsibility = matched_responsibility_evidence / total_core_responsibilities * 100
score_project = avg(top3 project_to_jd_similarity) * 100
score_experience = min(candidate_years / required_years, 1.0) * 100
score_education = education_level_ratio * 100
score_domain = domain_keyword_hit_ratio * 100
score_expression = evidence_clarity_score_from_llm

rule_score =
0.30*score_required_skill +
0.20*score_responsibility +
0.10*score_experience +
0.05*score_education +
0.10*score_domain

semantic_score = 0.60*score_project + 0.40*score_responsibility_semantic

llm_judge_score =
0.40*evidence_sufficiency +
0.30*task_fit +
0.30*credibility
```

分数解释必须配套输出：
- 强匹配项
- 弱匹配项
- 缺口项
- “有经历但表达弱”的潜在加分项

#### 2.3 简历优化
目标不是“替用户造一份厉害简历”，而是“把已有事实表达成更贴合目标岗位的版本”。

处理链路：
1. 诊断
   - 读取 `match_report`
   - 产出 `rewrite_tasks`
   - 每个任务包含：
     - 缺口点
     - 目标 section
     - 可用证据锚点
     - 禁止扩写说明
2. 证据绑定
   - 从原简历中为每个任务选 1-3 个 anchor bullets。
   - 若找不到 anchor，只能给“建议补充信息”，不能自动生成内容。
3. 改写
   - LLM 输入：原始 bullet、目标 JD 关键词、允许强调的能力。
   - 输出：更贴近岗位的表达版本。
   - 只允许做：
     - 重排顺序
     - 强调结果
     - 把职责表述映射到 JD 话术
     - 合并重复 bullet
   - 禁止做：
     - 新增不存在的项目
     - 编造成果数字
     - 引入未出现的技术栈
4. 校验
   - 对比改写前后 token diff。
   - 若新增实体词、数字、公司名、技术词不在证据白名单中，则判为高风险。
5. 生成最终简历
   - 生成 `draft_sections`
   - 允许用户手动编辑
   - 应用后生成新版本 resume

约束 LLM 不虚构的核心做法：
- Prompt 明确声明你是“rewrite-only editor”，不是 content creator。
- 输入中传入 `allowed_facts` 白名单：公司名、岗位名、时间、技术栈、项目名、已有数字。
- 输出要求每条 bullet 附 `evidence_refs`，指向原文片段或 section anchor。
- 结果再过一层 rule validator：
  - 新数字检测
  - 新实体检测
  - 新技能检测
  - 时间线冲突检测

优化的推荐产物：
- `diagnosis_json`
- `rewrite_tasks`
- `draft_sections_json`
- `fact_check_report_json`

#### 2.4 模拟面试
目标不是做聊天机器人，而是做“岗位驱动、基于你简历弱点和亮点的结构化面试训练”。

处理链路：
1. 面试规划
   - 输入：
     - 简历结构化结果
     - JD 结构化结果
     - `match_report.interview_blueprint_json`
     - 可选：`optimization_session`
   - 输出：
     - question plan
     - focus areas
     - rubric
     - ending rule
2. 动态出题
   - 问题来源分三类：
     - 优势展开题：追问最强证据
     - 缺口核验题：问 JD 要求但简历证据弱的点
     - 行为/通用题：稳定兜底
3. 追问逻辑
   - 若答案相关但空泛：追问细节、行动、结果、指标。
   - 若答案偏题：切下一个问题。
   - 每个主问题最多 1 次追问，防止会话拖垮。
4. 评分逻辑
   - 每轮评分维度：
     - relevance
     - specificity
     - evidence
     - structure
     - communication
   - 每轮 0-5 分，累计加权到总分。
5. 复盘总结
   - 对每题给：
     - 答得好的地方
     - 缺了什么
     - 更好的回答方向
   - 最终给：
     - 总分
     - readiness signal
     - 针对简历的反向补强建议

结构化面试报告建议：
```json
{
  "overall_score": 78,
  "dimension_scores": {
    "relevance": 82,
    "specificity": 65,
    "evidence": 70,
    "structure": 80,
    "communication": 75
  },
  "strengths": [],
  "weaknesses": [],
  "question_reviews": [
    {
      "question_text": "",
      "what_went_well": "",
      "what_was_missing": "",
      "better_answer_direction": ""
    }
  ],
  "follow_up_tasks": [
    {
      "title": "",
      "instruction": "",
      "target_section": "work_experience_or_projects"
    }
  ],
  "job_readiness_signal": {
    "status": "tailoring_needed|interview_ready|ready_to_apply",
    "reason": ""
  }
}
```

### 3. 算法方案总原则
- 规则负责底座：抽取、校验、硬约束、事实防伪。
- 向量负责找证据：把 JD 要求和简历片段建立候选映射。
- 重排负责精排：找出最能支撑某项要求的具体经历。
- LLM 负责语义评估和表达增强：解释匹配原因、生成优化草案、控制追问与复盘。
- 任何 LLM 输出都不能直接当事实源，只能当“建议层”或“评审层”。

### 4. 工程落地
#### 4.1 模块 I/O 与服务边界
- Resume Parse
  - Input: file bytes
  - Output: `raw_text`, `structured_json`, `parse_artifacts_json`
  - Core services: `storage`, `resume_parser`, `resume_ai`
- Match Report
  - Input: `resume structured_json`, `job structured_json`
  - Output: `scorecard_json`, `gap_json`, `evidence_map_json`, `tailoring_plan_json`, `interview_blueprint_json`
  - Core services: `job_parser`, `match_engine`, `match_ai`
- Resume Optimization
  - Input: `match_report`, `resume structured_json`
  - Output: `selected_tasks`, `draft_sections`, `fact_check_report`
  - Core services: `resume_optimizer`, `optimizer_ai_rewriter`, `fact_checker`
- Mock Interview
  - Input: `match_report`, `resume`, optional `optimization_session`
  - Output: `plan_json`, `turn evaluation`, `review_json`
  - Core services: `mock_interview`, `mock_interview_ai`

#### 4.2 适合工作流编排的地方
- 简历解析：上传后异步跑 OCR/解析/AI 校正/入库，适合后台 job。
- 岗位匹配：生成报告时间较长，适合异步 job。
- 简历优化：生成草案和 fact check 可以做成 job。
- 模拟面试：
  - 规划阶段适合单次 LLM 调用。
  - 每轮答题后评估适合同步单次 LLM 调用。

#### 4.3 MVP 路线
Phase 1:
- 只支持文本型 PDF
- 简历规则解析 + LLM 校正
- JD 规则解析
- 规则匹配 + LLM 评审
- 优化只改 summary / work_experience / projects
- 面试只做文字问答，一题一追问

Phase 2:
- 支持 DOCX、图片和 OCR
- 加入 embedding 检索和 evidence map
- 优化加入事实校验报告
- 面试加入模式切换（behavioral / technical / hr fit）

Phase 3:
- 多版本简历对比
- 用户补充缺失事实后自动重算匹配
- 面试语音化 / 实时化

#### 4.4 单人开发优先级
1. 先把“结构化事实层”做稳：简历解析 + JD 解析。
2. 再把“可解释匹配报告”做出来，因为这是优化和面试的上游。
3. 然后做“基于证据锚点的简历优化”，一定先上事实约束。
4. 最后做“结构化模拟面试”，因为它天然依赖前两步产物。

具体排序：
1. 统一 resume/job schema
2. 跑通解析异步链路
3. 输出 match report 六类 JSON
4. 基于 `tailoring_plan_json` 生成优化草案
5. 基于 `interview_blueprint_json` 生成面试计划和评审

MacBook Air M4 的现实建议：
- 不要训练模型。
- 用现成 OCR、embedding、LLM API。
- 向量检索先用 pgvector 或 SQLite + 本地小规模缓存，别一开始上重型检索平台。
- 后端先保证 traceable JSON 和 job status，前端先把闭环展示出来。
