# CareerPilot 简历解析与管理 / 简历中心模块说明文档

基于《简历解析与管理模块开发清单》《简历中心与岗位匹配使用教程》与后端 API 基线整合  

> **模块定位**  
> “简历解析与管理 / 简历中心”不是一个单纯的上传页，而是整个 CareerPilot 的简历数据基础设施模块。它的核心价值，是把用户的 PDF 简历转成可保存、可编辑、可复用的结构化简历资产，供岗位匹配、简历优化、模拟面试等模块继续消费。

## 一、文档目的

本文档用于简历中心模块说明，并补充你在沟通中提出的最新定位，以及后端 API 基线已经落地的实现现状。

文档重点解决三个问题：这个模块到底是什么、它当前应该具备哪些核心功能、以及现有后端接口已经支撑到了什么程度。

## 二、模块定位与边界

从产品定位上看，简历中心的本质不是“上传文件”，而是“沉淀简历结构化数据”。其核心闭环为：**上传 -> 解析 -> 展示 -> 校正 -> 复用**。

- **上传：** 用户在 `/dashboard/resume` 上传 PDF 简历，系统校验并保存到对象存储。
- **解析：** 系统抽取 PDF 原始文本，并输出标准 `structured_json` 结构。
- **展示：** 页面展示原始文本、解析状态、结构化结果与任务记录。
- **校正：** 用户可手动修正教育、工作、项目、技能等关键字段，并保存可用版本。
- **复用：** 修正后的结构化数据将被岗位匹配、简历优化、模拟面试等下游模块消费。

因此，这个模块更准确的角色是 CareerPilot 的“简历数据入口与数据中台”，而不是一个孤立的上传页。

## 四、核心业务闭环

围绕“上传 -> 解析 -> 展示 -> 校正 -> 复用”这个闭环，模块的业务职责可以拆成以下 5 个阶段：

### 4.1 上传与存储

- 仅支持 PDF 简历上传。
- 进行文件格式与大小校验。
- 将文件保存到对象存储（MinIO），保留 bucket 与 object key。
- 支持列表展示、删除和下载原文件。

### 4.2 解析与结构化

- 从文本型 PDF 中提取 `raw_text`。
- 使用规则引擎将非结构化文本转换为 `structured_json`。
- 第一版重点抽取 `basic_info`、`education`、`work_experience`、`projects`、`skills`、`certifications`。
- `skills` 进一步拆分为 `technical`、`tools`、`languages`，方便岗位匹配复用。

### 4.3 任务状态管理

- 为每次解析建立独立 parse job。
- 追踪 `pending`、`processing`、`success`、`failed` 等状态。
- 保存错误原因、开始结束时间与尝试次数。
- 支持失败后手动重新解析，并保留完整历史。

### 4.4 结构化展示与人工校正

- 页面展示简历基础信息、原始文本、结构化字段与解析记录。
- 用户可直接编辑教育经历、工作经历、项目经历、技术技能、个人摘要等关键内容。
- 保存后更新 `structured_json`，并递增 `latest_version`。

### 4.5 下游模块复用

- 岗位匹配模块直接消费解析成功或人工修正后的简历版本。
- 后续简历优化、投递追踪、模拟面试都可以基于相同结构继续扩展。

## 五、数据设计与结构化输出

| 数据对象 | 关键字段 | 说明 |
| --- | --- | --- |
| `resumes` | `file_name`, `file_url`, `raw_text`, `structured_json`, `parse_status`, `latest_version` | 保存简历文件元信息、原始文本、结构化结果与版本 |
| `resume_parse_jobs` | `status`, `attempt_count`, `error_message`, `started_at`, `finished_at` | 记录每次解析执行历史，支持失败重试与状态追踪 |
| `structured_json.basic_info` | `name`, `email`, `phone`, `location`, `summary` | 基础身份与联系方式信息 |
| `structured_json.sections` | `education`, `work_experience`, `projects`, `skills`, `certifications` | 供岗位匹配等模块继续消费的结构化字段 |

建议采用统一的 JSON 结构作为模块对外的简历数据标准，其中 `basic_info` 用于基础身份信息，`sections` 用于教育、工作、项目、技能和证书等可被复用的业务字段。

> **推荐结构化规范**  
> `{ basic_info, education, work_experience, projects, skills:{ technical, tools, languages }, certifications }`。这一结构既便于前端编辑，也便于岗位匹配引擎消费。

## 六、解析规则设计建议

在技术路线层面，开发清单强调第一版优先采用“规则引擎 + 正则 + 关键词 section heading 识别”的方式，而不是一开始就依赖 AI。这样做的目标是让规则可解释、结果可控、后续可扩展。

- **姓名：** 优先从前几行中识别非邮箱、非电话、长度较短的候选文本。
- **邮箱：** 使用标准邮箱正则匹配。
- **电话：** 使用宽松电话正则匹配，兼容手机号与国际号码。
- **地点：** 通过城市词典或 `remote` 等关键词识别。
- **教育 / 工作 / 项目 / 技能 / 证书：** 优先基于 section heading 切分，再抽取对应段落内容。
- **技能分类：** 按关键词映射到技术技能、工具技能、语言能力。
- **去重：** 同字段内部做大小写归一化与空格清洗，保留首次出现顺序。

如果后续需要把解析能力单独封装为 Python 处理脚本，这套规则同样可以直接沉淀到独立 parser service 中；而当前仓库已经给出了后端 API 化的主干路径，因此更推荐“脚本能力内聚到服务层，接口对外统一暴露”的实现方式。

## 七、页面与交互形态

前端建议并已在说明中统一为一个真实业务页：`/dashboard/resume`。第一版不需要拆成多个子页面，而是做成列表 + 详情 + 编辑的工作台。

- **左栏：** 上传卡片 + 简历列表，展示文件名、上传时间、解析状态、版本号。
- **右栏顶部：** 状态徽章、文件信息、重试解析、下载原文件、删除、保存修正。
- **右栏中部：** 原始文本预览、失败原因、解析任务历史。
- **右栏底部：** 结构化编辑器，支持 `basic_info`、`education`、`work_experience`、`projects`、`skills`、`certifications` 编辑。

## 八、后端 API 基线与当前实现现状

根据已提供的 `resumes.py`，简历中心的后端路由基线已经比较完整，说明模块已经从“纯规划”进入“可联调实现”的阶段。当前基线具备以下特征：

- 路由前缀统一为 `/resumes`，符合 REST 风格。
- 上传接口在创建 resume 后，会通过 `BackgroundTasks` 异步触发 `process_resume_parse_job`。
- 列表、详情、删除、下载链接、解析任务历史、手动重试解析、保存结构化结果都已有路由定义。
- 接口统一使用 `success_response` 与 `ApiSuccessResponse` 包装，保持与当前仓库风格一致。
- 当前用户通过 `Depends(get_current_user)` 注入，具备按用户隔离的权限基础。

| 方法 | 路径 | 用途 | 现状 |
| --- | --- | --- | --- |
| `POST` | `/resumes/upload` | 上传 PDF 并触发后台解析 | 已实现，解析任务通过 `BackgroundTasks` 启动 |
| `GET` | `/resumes` | 获取当前用户简历列表 | 已实现 |
| `GET` | `/resumes/{resume_id}` | 获取简历详情与结构化信息 | 已实现 |
| `DELETE` | `/resumes/{resume_id}` | 删除简历及对象存储文件 | 已实现 |
| `GET` | `/resumes/{resume_id}/download-url` | 生成预签名下载链接 | 已实现 |
| `GET` | `/resumes/{resume_id}/parse-jobs` | 查看解析任务历史 | 已实现 |
| `POST` | `/resumes/{resume_id}/parse` | 手动重试解析 | 已实现 |
| `PUT` | `/resumes/{resume_id}/structured` | 保存人工修正后的 `structured_json` | 已实现 |
