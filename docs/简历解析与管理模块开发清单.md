# 简历解析与管理模块开发清单

> 项目：职点迷津（CareerPilot）
> 模块范围：简历上传、解析、结构化、人工校正
> 生成日期：2026-03-15
> 目标：基于当前已完成的登录、个人资料，跑通"上传简历 → 自动解析 → 结构化展示 → 人工校正"的完整工作流。

---

## 1. 这次开发要解决什么问题

当前项目已经具备：

- 用户登录注册
- 个人资料与求职偏好

但还缺少用户上传简历的核心能力：

1. 用户没有地方上传 PDF 简历。
2. 系统无法自动从 PDF 中抽取文本和结构化信息。
3. 后续的岗位匹配、简历优化、模拟面试都缺少简历数据输入。

所以简历解析与管理模块的目标很明确：

1. 让用户能上传 PDF 格式的简历。
2. 让系统自动抽取 PDF 文本并结构化。
3. 让用户能查看和人工校正结构化结果。

---

## 2. 本次开发范围

### 2.1 本次必须做

- PDF 简历上传（MinIO 存储）
- PDF 文本自动抽取
- 结构化信息抽取（姓名、邮箱、电话、地点、教育、工作经历、项目、技能、证书）
- 解析状态追踪与失败重试
- 结构化结果人工校正与保存
- 历史版本管理
- 原始文件下载

### 2.2 本次先不做

- Word / TXT 等格式简历支持
- AI 增强的结构化抽取（如智能补全、纠错）
- 多份简历批量上传
- 简历模板推荐
- 简历质量评分
- 简历导出为 PDF / Word

说明：
第一版的重点是把核心工作流跑通。结构化抽取先用规则引擎（关键词 + 正则），后续再逐步接 AI 增强。

## 3. 模块落地顺序

建议按下面 4 个阶段推进，避免同时开太多口子。

### 阶段 A：数据层

目标：先把简历和解析任务存下来。

要完成：

1. 新增 `resumes` 表（含 `structured_json` 字段）
2. 新增 `resume_parse_jobs` 表（追踪解析历史）
3. 新增 Resume 的模型、Schema、Service、Router、测试
4. 完成 `/dashboard/resume` 的简历列表 + 上传表单

### 阶段 B：上传与存储

目标：让用户能上传 PDF，并安全存储。

要完成：

1. 接入 MinIO 对象存储
2. 完成文件校验（格式、大小）
3. 完成文件重命名与安全存储
4. 完成下载链接生成

### 阶段 C：解析与结构化

目标：把 PDF 文本变成可操作的结构化数据。

要完成：

1. PDF 文本抽取（pypdf）
2. 结构化 JSON 规范定义
3. 规则引擎抽取（姓名、邮箱、电话、地点、教育、工作、项目、技能、证书）
4. 解析任务异步执行与状态追踪
5. 失败重试机制

### 阶段 D：前端展示与人工校正

目标：让用户能查看和修正结构化结果。

要完成：

1. 简历列表展示（状态、版本、时间）
2. 详情面板（原始文本预览、解析任务记录）
3. 结构化编辑器（各字段可编辑）
4. 保存与版本管理

## 4. 数据表设计

### 4.1 `resumes` 表

用途：保存用户上传的简历文件元信息、原始文本、结构化结果。

建议字段如下：

| 字段名               | 类型         | 说明             | 备注                                      |
| -------------------- | ------------ | ---------------- | ----------------------------------------- |
| `id`                 | UUID         | 简历主键         | 主键                                      |
| `user_id`            | UUID         | 所属用户         | FK -> `users.id`                          |
| `file_name`          | VARCHAR(255) | 原始文件名       | 必填                                      |
| `file_url`           | TEXT         | MinIO 文件 URL   | 必填                                      |
| `storage_bucket`     | VARCHAR(120) | 存储桶名称       | 必填                                      |
| `storage_object_key` | TEXT         | 存储对象键       | 必填、唯一                                |
| `content_type`       | VARCHAR(120) | 文件类型         | 默认 `application/pdf`                    |
| `file_size`          | INTEGER      | 文件大小（字节） | 必填                                      |
| `parse_status`       | VARCHAR(20)  | 解析状态         | `pending / processing / success / failed` |
| `parse_error`        | TEXT         | 解析失败原因     | 可空                                      |
| `raw_text`           | TEXT         | 抽取的原始文本   | 可空                                      |
| `structured_json`    | JSON         | 结构化结果       | 可空                                      |
| `latest_version`     | INTEGER      | 当前版本号       | 默认 1                                    |
| `created_at`         | TIMESTAMP    | 创建时间         | 复用当前 mixin 风格                       |
| `updated_at`         | TIMESTAMP    | 更新时间         | 复用当前 mixin 风格                       |
| `created_by`         | UUID         | 创建人           | 复用当前 mixin 风格                       |
| `updated_by`         | UUID         | 更新人           | 复用当前 mixin 风格                       |

建议索引：

- `idx_resumes_user_id`
- `idx_resumes_parse_status`
- `idx_resumes_created_at`

说明：

- `storage_object_key` 用唯一约束，避免重复上传同一文件。
- `structured_json` 用 JSON 类型，方便后续扩展字段。
- `latest_version` 用于追踪人工修正历史。

### 4.2 `resume_parse_jobs` 表

用途：追踪每次解析任务的执行历史（支持重试）。

建议字段如下：

| 字段名          | 类型        | 说明     | 备注                                      |
| --------------- | ----------- | -------- | ----------------------------------------- |
| `id`            | UUID        | 任务主键 | 主键                                      |
| `resume_id`     | UUID        | 关联简历 | FK -> `resumes.id`                        |
| `status`        | VARCHAR(20) | 任务状态 | `pending / processing / success / failed` |
| `attempt_count` | INTEGER     | 尝试次数 | 默认 0                                    |
| `error_message` | TEXT        | 错误信息 | 可空                                      |
| `started_at`    | TIMESTAMP   | 开始时间 | 可空                                      |
| `finished_at`   | TIMESTAMP   | 结束时间 | 可空                                      |
| `created_at`    | TIMESTAMP   | 创建时间 | 复用当前 mixin 风格                       |
| `updated_at`    | TIMESTAMP   | 更新时间 | 复用当前 mixin 风格                       |
| `created_by`    | UUID        | 创建人   | 复用当前 mixin 风格                       |
| `updated_by`    | UUID        | 更新人   | 复用当前 mixin 风格                       |

建议索引：

- `idx_resume_parse_jobs_resume_id`
- `idx_resume_parse_jobs_status`
- `idx_resume_parse_jobs_created_at`

说明：

- 每次重试解析都新建一条记录，保留完整历史。
- `attempt_count` 记录重试次数，方便后续限流。

## 5. 后端接口设计

接口命名要尽量贴合当前仓库风格，优先走 REST 风格，响应继续复用当前统一响应格式。

| ------ | ---- | -------- |

```json
  "user_id": "uuid",

  "file_url": "minio://bucket/resumes/user-id/resume-id/张三-前端开发.pdf",
  "storage_bucket": "career-pilot",
  "storage_object_key": "resumes/user-id/resume-id/张三-前端开发.pdf",
  "content_type": "application/pdf",
  "file_size": 1024000,
  "parse_status": "pending",
  "parse_error": null,
  "raw_text": null,
  "structured_json": null,
  "latest_version": 1,
  "created_at": "2026-03-15T10:00:00Z",
  "updated_at": "2026-03-15T10:00:00Z",
  "latest_parse_job": {
    "id": "uuid",
    "status": "pending",
    "attempt_count": 0,
    "error_message": null,
    "started_at": null,
    "finished_at": null,
    "created_at": "2026-03-15T10:00:00Z",
```

```json
    "updated_at": "2026-03-15T10:05:00Z",

```

#### 3. 获取简历详情

- 方法：`GET /resumes/{resume_id}`
- 作用：获取简历完整信息（含结构化结果）

#### 4. 删除简历

- 方法：`DELETE /resumes/{resume_id}`
- 作用：删除简历（同时删除 MinIO 中的文件）

响应：

```json
{
  "message": "Resume deleted successfully"
}
```

- 方法：`POST /resumes/{resume_id}/parse`
- 作用：当解析失败或用户希望重新解析时触发

响应：同上传接口（新建一条 parse job）

#### 6. 获取解析任务历史

- 方法：`GET /resumes/{resume_id}/parse-jobs`
- 作用：查看该简历的所有解析记录

响应：

```json
[
  {
    "id": "uuid",
    "status": "success",
    "attempt_count": 2,
    "error_message": null,
    "started_at": "2026-03-15T10:00:00Z",
```

- 方法：`PUT /resumes/{resume_id}/structured`
- 作用：用户修正结构化结果后保存

请求体：

```json
{
  "structured_json": {
    "basic_info": {
      "name": "张三",
      "email": "zhangsan@example.com",
      "phone": "13800138000",
      "location": "上海",

    },
    "education": ["复旦大学 - 计算机科学与技术"],
```

## 8. 结构化抽取规则

第一版一定要坚持两个原则：

1. 规则必须可解释（能说出为什么抽到这个）。
2. 先规则化，再考虑 AI 增强。

### 8.1 结构化 JSON 规范

```json
{
  "basic_info": {
    "name": "张三",
    "email": "zhangsan@example.com",
    "phone": "13800138000",
    "location": "上海",
    "summary": "5年前端开发，擅长 React..."
  },
  "education": ["复旦大学 - 计算机科学与技术 - 本科 - 2015-2019"],
  "work_experience": [
    "字节跳动 - 前端工程师 - 2021-2024",
    "腾讯 - 实习前端工程师 - 2019-2021"
  ],
  "projects": ["简历平台重构项目 - 负责前端架构升级 - React + TypeScript"],
  "skills": {
    "technical": ["React", "Vue", "TypeScript", "Node.js"],
    "tools": ["Git", "Docker", "Figma", "Jira"],
    "languages": ["English", "CET-6"]
  },
  "certifications": ["英语六级", "PMP", "阿里云认证"]
}
```

说明：

- `basic_info` 是基础信息，尽量抽全。
- `education` / `work_experience` / `projects` 用字符串数组，每条字符串保留原文关键信息。
- `skills` 拆成 `technical` / `tools` / `languages` 三类，方便后续匹配。
- `certifications` 保留原文。

### 8.2 字段来源与抽取规则

#### A. 基础信息

**姓名：**

- 规则：取文本前 5 行中，非邮箱/电话、长度 ≤ 32 的第一行。
- 证据：多数简历第 1 行是姓名。

**邮箱：**

- 规则：正则匹配 `[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}`
- 证据：标准邮箱格式。

**电话：**

- 规则：正则匹配 `(?:\+?\d[\d\s-]{7,}\d)`
- 证据：常见手机号/电话格式。

**地点：**

- 规则：关键词匹配（北京、上海、广州、深圳、杭州、成都、武汉、苏州、南京、remote）
- 证据：简历通常会写工作地点。

**摘要：**

- 规则：取第 2-8 行中，非邮箱/电话、非 section heading 的前 3 行。
- 证据：多数简历在姓名后有简短介绍。

#### B. 教育经历

- 规则：匹配 section heading（教育、教育经历、教育背景、education）
- 提取：该 section 下所有非空行
- 证据：简历通常有明确的教育 section。

#### C. 工作经历

- 规则：匹配 section heading（工作经历、实习经历、experience、professional experience）
- 提取：该 section 下所有非空行
- 证据：简历通常有明确的工作经历 section。

#### D. 项目经历

- 规则：匹配 section heading（项目经历、项目经验、项目背景、projects、project experience）
- 提取：该 section 下所有非空行
- 证据：简历通常有明确的项目 section。

#### E. 技能

- 规则：匹配 section heading（技能、专业技能、skills、skill、tech stack）
- 提取：该 section 下所有非空行
- 分类：用关键词匹配技术 / 工具 / 语言
  - 技术：Python、Java、JavaScript、TypeScript、React、Vue、Next.js、FastAPI、Django、SQL、PostgreSQL、MySQL、Redis、Excel、Power BI、Tableau
  - 工具：Git、Docker、Figma、Postman、Jira、Linux、Photoshop
  - 语言：英语、英文、日语、韩语、English、Japanese、Korean

#### F. 证书与奖项

- 规则：匹配 section heading（证书、证书奖项、certifications、awards）
- 提取：该 section 下所有非空行
- 证据：简历通常有明确的证书 section。

### 8.3 去重策略

- 同一字段内去重（如 skills.technical），保留首次出现顺序。
- 去重时做大小写归一化 + 空格清理。

## 13. 最终结论

这次简历解析与管理模块的核心不是"做一个新页面"，而是建立简历数据的基础设施：

- 简历是输入
- 结构化结果是中间层
- 岗位匹配、简历优化、投递追踪、模拟面试都会从这里继续长出来

所以开发时一定坚持这两个原则：

1. 先把规则化、可解释、可保存的结构化结果做出来。
2. 先保证闭环，再逐步引入 AI 增强。

只要按这份清单执行，简历解析与管理模块就会是《职点迷津》真正进入"产品能力阶段"的第一个核心模块。

本项目的简历解析功能暂不采用后端接口模式，而是作为一套独立运行的 python 处理脚本开发。输入：PDF 格式的简历文件；输出：标准 JSON 格式的结构化数据；当前最关键的问题：中间处理过程如何具体落地？需要明确以下三点：（1）提取逻辑：如何从 PDF 文件中准确提取原始文本？（2）解析规则：如何将非结构化文本转换为标准 JSON 字段？（3）规则设计：各字段的识别规则、优先级、容错机制如何安排？要求开发语言为 Python；参考来源：可联网搜索，借鉴 GitHub 上优秀的开源简历解析项目；代码风格：逻辑清晰、规则可配置、便于后续扩展。
