# CareerPilot 模块开发约定

> 适用阶段：节点 1 基础治理补齐后
> 目标：后续新增 `resume`、`jobs`、`applications`、`interviews` 等模块时，不再重复搭请求层、错误处理、鉴权和页面状态规范

---

## 1. 后端约定

### 1.1 统一响应格式

成功响应：

```json
{
  "success": true,
  "data": {},
  "meta": {
    "request_id": "uuid",
    "timestamp": "2026-03-14T16:00:00Z"
  }
}
```

失败响应：

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": []
  },
  "meta": {
    "request_id": "uuid",
    "timestamp": "2026-03-14T16:00:00Z"
  }
}
```

约定说明：

- `request_id` 由服务端统一生成并写入响应头 `X-Request-ID`
- 前端所有错误提示优先读取 `error.message`
- 调试和排障时优先记录 `request_id`

### 1.2 错误码使用原则

- 参数校验失败：`VALIDATION_ERROR`
- 未登录/无 token：`AUTH_MISSING_TOKEN`
- token 过期：`AUTH_TOKEN_EXPIRED`
- token 无效：`AUTH_TOKEN_INVALID`
- token 已注销：`AUTH_TOKEN_REVOKED`
- 登录失败：`AUTH_INVALID_CREDENTIALS`
- 账号禁用：`AUTH_USER_DISABLED`
- 数据冲突：`USER_ALREADY_EXISTS`、`CONFLICT`
- 系统异常：`INTERNAL_SERVER_ERROR`

### 1.3 路由与模块结构

当前后端仍采用“按层拆分”的目录结构，后续新增模块时遵循以下分层：

```text
apps/api/app/
├── api/routes/           # 对外接口
├── models/               # SQLAlchemy 模型
├── schemas/              # 请求/响应模型
├── services/             # 业务逻辑
├── core/                 # 配置、错误、响应、日志、安全
└── db/                   # 会话与基类
```

新模块最少需要补齐：

```text
api/routes/{module}.py
models/{module}.py
schemas/{module}.py
services/{module}.py
tests/test_{module}.py
alembic/versions/..._{module}.py
```

推荐命名：

- 简历中心：`resume`
- 岗位匹配：`job`
- 投递追踪：`application`
- 模拟面试：`interview`

### 1.4 审计字段规范

业务表优先使用以下基础字段：

- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

当前节点已在 `user_profiles` 上落地，后续新业务表应默认复用这套字段。

### 1.5 健康检查规范

当前统一提供：

- `GET /health`
- `GET /health/readiness`
- `GET /health/version`

约定：

- `health` 只反映服务进程是否存活
- `readiness` 用于检查数据库和关键依赖是否就绪
- `version` 用于前后端联调确认运行环境

### 1.6 认证演进策略

当前生产骨架仍使用：

- access token
- 前端本地存储
- `/auth/logout` + token blocklist

这是当前阶段的可用方案，但不是长期终态。升级口已经预留在后端配置和 token 结构里，后续建议演进为：

1. access token + refresh token 双 token
2. refresh token 放 `httpOnly cookie`
3. access token 短时效
4. 前端静默刷新

在没切换前，所有新模块都继续依赖当前 Bearer Token 即可。

## 2. 前端约定

### 2.1 API 分层

以后页面和组件不要直接写 `fetch`，统一走：

```text
apps/web/src/lib/api/
├── client.ts             # 请求底座、错误解析
├── contracts.ts          # 响应契约类型
└── modules/
    ├── auth.ts
    ├── profile.ts
    └── future-module.ts
```

使用方式：

- 请求底座统一在 `client.ts`
- 每个业务模块单独一个 `modules/*.ts`
- 页面只调用模块函数，不关心响应包裹结构

### 2.2 路由守卫规范

当前已经落地两类守卫：

- `ProtectedRoute`
  - 用于 Dashboard、后续所有需登录页面
  - 未登录自动跳转 `/login?next=当前路径`
- `GuestRoute`
  - 用于 `/login`、`/register`
  - 已登录自动回到工作台

约定：

- 新增 dashboard 页面默认属于受保护页面
- 需要匿名访问的页面不得复用 dashboard 布局

### 2.3 页面状态规范

页面级状态统一复用：

- `PageLoadingState`
- `PageEmptyState`
- `PageErrorState`

约定：

- 首屏数据加载必须有 loading 态
- 空列表或未配置状态必须有 empty 态
- 请求失败必须有 error 态和重试动作

### 2.4 Dashboard 布局复用规则

Dashboard 公共壳层已经由：

- 左侧导航
- 顶部 Header
- 受保护路由
- 页面内容区

组成。

后续业务页面只负责“内容区”：

- 不重复写 sidebar
- 不重复写登录判断
- 不重复写通用头部
- 不直接改动全局布局 unless 是跨模块通用能力

### 2.5 Profile / Resume / Jobs 页面开发标准

每个真实业务页建议遵循下面的顺序：

1. 页面顶区：说明当前模块价值和路由位置
2. 首屏加载：请求接口并展示 loading
3. 数据失败：显示 error + retry
4. 表单/列表主区：承载真实业务交互
5. 右侧说明区：解释该模块和后续模块的关系

## 3. 测试与验收约定

### 3.1 后端

- 每新增一个路由模块，至少补一组 happy path 测试
- 认证相关接口必须覆盖未授权场景
- 迁移新增表后必须至少跑一次 pytest

### 3.2 前端

- 提交前至少执行 `npm run lint`
- 有路由结构变动时必须执行 `npm run build`
- 首屏页面至少验证 loading / success / error 三种状态

## 4. 当前节点落地结果

节点 1 已落地：

- 统一响应格式
- 统一错误码
- request id 与请求日志
- health / readiness / version 接口
- 前端 API 分层
- 路由守卫
- 页面级状态组件
- Dashboard 复用规则

节点 2 已落地：

- `user_profiles` 数据表
- 个人资料读取与更新接口
- `/dashboard/profile` 真实表单页
- 昵称、求职方向、目标城市、期望岗位可写入数据库

这份文档从现在开始就是后续模块开发的默认标准。
