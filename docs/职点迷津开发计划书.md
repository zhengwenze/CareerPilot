# 《职点迷津》开发计划书

> 版本：v1.0
> 生成日期：2026-03-14
> 适用范围：基于当前仓库真实代码现状制定的 V1 执行计划

## 一、项目基本信息

- 项目名称：职点迷津（CareerPilot）
- 产品定位：面向求职者的 AI 求职工作台，提供简历管理、岗位匹配、简历优化、模拟面试、投递追踪和复盘分析
- 项目类型：全栈 Web 应用
- 当前技术路线：Next.js 16 + React 19 + Tailwind CSS 4 + shadcn/ui + FastAPI + SQLAlchemy + Alembic + PostgreSQL + Redis + MinIO
- 当前完成度判断：约 15% - 20%
- 当前已完成范围：用户注册、登录、退出、当前用户查询；前端认证状态恢复；Dashboard 路由壳层与占位页

## 二、当前项目现状解析

### 2.1 前端现状

当前前端已经具备一个可继续扩展的控制台骨架：

- 已完成认证页面与认证上下文
  - `apps/web/src/components/auth-provider.tsx`
  - `apps/web/src/components/auth-form.tsx`
  - `apps/web/src/lib/auth.ts`
- 已完成 App Router 路由组织
  - `apps/web/src/app/login/page.tsx`
  - `apps/web/src/app/register/page.tsx`
  - `apps/web/src/app/(dashboard)/layout.tsx`
- 已完成 Dashboard 左侧导航和各业务页路由占位
  - `apps/web/src/components/layout/app-sidebar.tsx`
  - `apps/web/src/config/nav-config.ts`
  - `apps/web/src/app/(dashboard)/dashboard/*/page.tsx`

前端当前优点：

- 认证链路已经打通，登录后能进入工作台
- 路由结构清晰，后续业务页可按模块逐个接入
- 主题变量和基础 UI 组件已经形成统一风格

前端当前缺口：

- 所有业务页面仍为占位页，没有真实数据和交互
- 没有简历上传、岗位录入、投递记录、面试练习等核心页面
- 没有表单体系、列表筛选、详情页、状态管理规范
- Token 目前保存在 `localStorage`，安全性只能满足开发期，不建议作为长期正式方案
- 还没有建立前端 API 分层、缓存策略、错误边界和埋点体系

### 2.2 后端现状

当前后端是一个认证服务骨架，代码质量和结构都适合继续扩展：

- 已完成 FastAPI 应用入口和基础异常处理
  - `apps/api/app/main.py`
- 已完成认证接口
  - `apps/api/app/api/routes/auth.py`
  - `POST /auth/register`
  - `POST /auth/login`
  - `POST /auth/logout`
  - `GET /auth/me`
- 已完成用户模型与迁移
  - `apps/api/app/models/user.py`
  - `apps/api/alembic/versions/20260314_0001_create_users_table.py`
- 已完成密码哈希、JWT 签发、Redis 黑名单登出
  - `apps/api/app/core/security.py`
  - `apps/api/app/services/token_blocklist.py`
- 已完成后端测试
  - `apps/api/tests/test_auth.py`

后端当前优点：

- 分层清楚，后面新增服务、模型、路由不会推倒重来
- Alembic 已经成为数据库结构来源，方向正确
- 认证流程有基础测试覆盖，能作为后续接口开发模板

后端当前缺口：

- 目前只有 `users` 一张真实业务表，所有求职核心数据尚未开始建模
- 没有简历、JD、匹配报告、投递记录、面试会话、评测结果等数据模型
- 没有文件上传、对象存储、异步任务、AI 调用、流式输出
- 没有权限管理、刷新令牌、限流、审计日志、操作追踪
- 没有面向业务模块的 service/repository/router 拆分规范文档

### 2.3 基础设施现状

仓库已经准备了未来扩展所需的基础设施，但大多还未接入业务：

- `docker-compose.middleware.yml`
  - PostgreSQL（镜像已带 pgvector）
  - Redis
  - MinIO
- `infra/sql/init.sql`
  - 仅作为本地引导脚本，不再是长期真源

当前判断：

- PostgreSQL 和 Redis 已经被认证模块实际使用
- MinIO 还没有接入简历上传
- pgvector 所在数据库镜像已经准备好，但向量字段、索引、检索流程尚未落地
- 仓库中还缺少 CI/CD、部署脚本、环境区分和监控告警方案

### 2.4 文档与真实代码的偏差

`DEV_DOCUMENT.md` 已经写出了完整产品愿景、ER 图和后续模块，但它更像“目标蓝图”，不是“当前实现说明”。

当前最大风险不是代码太少，而是：

- 目标文档很大，容易让开发顺序失焦
- 当前代码只完成认证，如果直接全面铺开，极易同时欠下前端、后端、AI、评测四类债务
- `packages/shared` 和 `packages/prompts` 目前为空，说明跨模块复用能力尚未开始建设

因此本计划书的核心策略是：

1. 不追求一次性做全所有模块。
2. 先跑通一个真实求职闭环，再扩展到完整工作台。
3. 每完成一个模块，必须同步补测试、补文档、补验收标准。

## 三、项目目标与 V1 范围重定义

### 3.1 V1 必须完成

围绕“一个用户可以完整完成一次 AI 求职训练闭环”来定义 V1：

1. 账户系统
   - 注册、登录、退出、当前用户
   - 基础个人信息
   - 求职偏好设置
2. 简历中心
   - PDF 上传
   - 简历列表
   - 解析状态查看
   - 结构化信息展示与人工校正
3. 岗位匹配
   - JD 录入
   - 结构化提取
   - 简历对 JD 的匹配评分
   - 差距说明
4. 简历优化
   - 基于目标 JD 输出修改建议
   - 输出分模块优化项
5. 模拟面试
   - 选择岗位方向后进入面试练习
   - 多轮问答
   - AI 点评
   - 历史记录保存
6. 投递追踪
   - 记录岗位、阶段、更新时间、备注
   - Dashboard 汇总待办和进展

### 3.2 V1 暂不做

- 支付和会员体系
- 企业端招聘后台
- 社区分享
- 多语言
- 复杂组织权限
- 原生移动 App

### 3.3 V1 成功标准

- 新用户从注册到进入工作台的流程稳定可用
- 用户能上传简历并看到结构化结果
- 用户能录入 JD 并看到匹配报告
- 用户能拿到一份可编辑的优化建议
- 用户能完成至少一轮模拟面试并查看复盘
- 用户能记录投递状态并在概览页看到进度
- 核心流程有基础测试、异常提示和可回归验证方式

## 四、建议采用的技术方案

### 4.1 保持当前技术栈，不建议重构

当前代码基础足够好，建议继续沿用：

- 前端：Next.js App Router + TypeScript + Tailwind CSS + shadcn/ui
- 后端：FastAPI + SQLAlchemy + Alembic + Pydantic
- 数据库：PostgreSQL
- 缓存与会话：Redis
- 文件存储：MinIO
- AI 能力：统一模型适配层，后续接 OpenAI 兼容接口

原因：

- 认证和工作台壳层都已经围绕这套栈写好了
- 现在换技术只会延后业务闭环
- Python 后端对后续解析、RAG、面试评分更友好

### 4.2 新增但必要的工程能力

建议在不推翻现有结构的前提下补齐以下能力：

- 前端新增：
  - React Hook Form + Zod，用于复杂表单
  - TanStack Query，用于业务数据请求、缓存和失效管理
  - 页面级 loading / empty / error 规范
- 后端新增：
  - 统一错误码枚举
  - 文件上传模块
  - AI Provider 适配层
  - 任务状态模型和后台异步任务执行机制
- 工程新增：
  - GitHub Actions 或本地统一脚本
  - `.env` 分层说明
  - E2E 测试

## 六、详细开发节点规划

> 说明：以下节点按照“单主开发 + AI 协作”节奏设计。若前后端两人并行，可将总周期从 8 周压缩到 5-6 周。

### 节点 1：基础治理补齐（1 周）

目标：把当前认证骨架升级为可持续开发的基础盘。

前端任务：

- 建立 API 模块分层，避免每个页面直接写 `fetch`
- 增加路由守卫与未登录跳转规范
- 补 loading、empty、error 的页面级组件
- 明确 Dashboard 公共布局可复用规则

后端任务：

- 统一 API 错误码和错误响应格式
- 增加 health、readiness、version 等基础接口规范
- 增加请求日志、trace id、基础审计字段
- 规划业务模块目录，例如 `resume/`, `jobs/`, `applications/`, `interviews/`

交付物：

- 模块开发约定文档
- 统一响应格式
- 前后端脚手架标准

验收标准：

- 新增业务模块时不再重复写认证、错误处理和请求封装
- 前后端 lint / build / pytest 稳定通过

### 节点 2：账户系统加固（0.5 周）

目标：在已有注册登录基础上补齐长期可用能力。

前端任务：

- 个人信息页从占位页改为真实表单
- 支持修改昵称、求职方向、目标城市、期望岗位

后端任务：

- 新增用户资料扩展字段或独立 `user_profiles` 表
- 增加个人资料读取与更新接口
- 评估是否补 refresh token，至少在计划内留下升级口

验收标准：

- 用户可以维护个人基础资料
- 认证信息和用户偏好可以被后续简历、面试、岗位模块复用

### 节点 3：简历中心 MVP（1.5 周）

目标：让用户真正能把简历放进系统。

前端任务：

- 把 `/dashboard/resume` 从占位页改成真实页面
- 支持文件上传、简历列表、状态标签、详情页
- 支持结构化简历结果展示和人工修正入口

后端任务：

- 设计 `resumes`、`resume_versions`、`resume_parse_jobs` 等表
- 接入 MinIO，完成 PDF 上传与元数据保存
- 完成简历上传接口、列表接口、详情接口、删除接口
- 完成简历解析任务状态流转

AI / 解析任务：

- 第一版先做稳定文本抽取和结构化骨架，不追求一次到位
- 明确结构字段：基本信息、教育、经历、项目、技能、证书

验收标准：

- 用户能上传 PDF 并看到“上传中 / 解析中 / 已完成 / 失败”状态
- 至少一份简历可以被成功解析并展示结构化内容

### 节点 4：JD 中心与岗位匹配 MVP（1.5 周）

目标：跑通“简历 + JD -> 匹配报告”。

前端任务：

- 把 `/dashboard/jobs` 改成真实列表 + 表单页
- 支持手动粘贴 JD、保存 JD、查看匹配结果
- 支持展示匹配得分、技能差距、关键词命中

后端任务：

- 设计 `job_descriptions`、`match_reports` 表
- 新增 JD 创建、列表、详情、删除接口
- 实现第一版规则匹配引擎
  - 关键词匹配
  - 技能覆盖度
  - 经验年限粗匹配
  - 学历/城市等基础条件命中
- AI 负责解释结果，不让模型直接裸打分

验收标准：

- 用户选择简历和一个 JD 后能生成匹配报告
- 匹配结果至少包含总分、优势项、短板项、行动建议

### 节点 5：简历优化与项目话术（1 周）

目标：让匹配结果转化成用户可执行的修改建议。

前端任务：

- 在简历详情或匹配报告页接入优化建议面板
- 支持按模块查看优化建议：摘要、经历、项目、技能
- 支持复制结果、标记已处理

后端任务：

- 设计 `optimization_reports`、`project_scripts` 表
- 生成面向特定 JD 的简历优化建议
- 生成项目介绍话术初版
- 保存建议历史，便于用户复看

验收标准：

- 用户从匹配报告一键进入优化建议
- 输出内容有结构，不是纯聊天文本

### 节点 6：投递追踪与工作台概览（1 周）

目标：把“分析能力”扩展成“求职过程管理能力”。

前端任务：

- 把 `/dashboard/applications` 和 `/dashboard/overview` 改成真实页面
- 提供投递列表、阶段状态、更新时间、备注
- 概览页展示待办、最近投递、最近生成的报告、待练习面试

后端任务：

- 设计 `applications`、`application_events` 表
- 新增投递新增、更新状态、备注、列表查询接口
- 新增概览聚合接口

验收标准：

- 用户至少能维护完整投递状态流
- 概览页不是静态卡片，而是基于真实数据汇总

### 节点 7：模拟面试 MVP（1.5 周）

目标：落地 V1 最核心的 AI 体验。

前端任务：

- 把 `/dashboard/interviews` 改成会话式页面
- 支持开始面试、连续回答、查看 AI 追问和点评
- 支持会话历史列表和单场详情

后端任务：

- 设计 `interview_sessions`、`interview_turns`、`interview_evaluations` 表
- 提供开始会话、提交回答、获取下一问、结束会话接口
- 如条件允许，使用 SSE 做流式输出；否则第一版先同步输出

AI / 规则任务：

- 先按岗位方向和简历内容生成问题
- 再做追问、评分、改进建议
- 所有评分必须保留证据片段

验收标准：

- 用户能完成一场最少 5 轮的模拟面试
- 会话结束后能看到总评和改进方向

### 节点 8：评测、观测、安全与上线准备（1 周）

目标：避免项目“能演示但不能稳定交付”。

前端任务：

- 增加 E2E 核心流程测试
- 增加页面埋点和失败提示规范

后端任务：

- 增加 AI 调用日志和任务状态日志
- 增加限流、文件大小校验、敏感接口保护
- 补齐错误码文档

基础设施任务：

- 编写生产环境 `.env` 模板
- 编写 Docker 部署文档
- 增加数据库备份和对象存储检查项

验收标准：

- 核心主流程可回归测试
- 本地部署和服务器部署文档可执行

## 八、前后端并行开发安排

### 8.1 可并行执行的任务

- 前端做 Dashboard 真实页面骨架时，后端可同步设计表结构和接口
- 前端先用假数据搭简历页 / JD 页骨架，后端完成接口后再联调
- 面试页的会话布局和评分面板可先做，AI 接口随后接入

### 8.2 必须串行的任务

- 数据表确认前，不要先写大量详情页逻辑
- 匹配报告字段没冻结前，不要提前做复杂图表
- 面试评分规则没定前，不要先做最终评分 UI

### 8.3 推荐分工

- 主控：你 + Codex，负责路线、验收、文档同步
- 后端 Agent：模型设计、迁移、接口、AI 编排
- 前端 Agent：页面、组件、交互、联调
- 测试 Agent：pytest、接口回归、E2E
- 文档 Agent：同步 `DEV_DOCUMENT.md`、API 文档、样式规范

- `interview_turns`
- `interview_evaluations`

接口优先级也按同样顺序推进，避免数据库先膨胀、业务却迟迟没有落地。

## 十、关键风险与对应策略

### 风险 1：目标太大，导致迟迟没有可用版本

策略：

- 以“求职闭环”而不是“功能总数”管理开发
- 每 1 周都要产出一个用户能点到的真实功能

### 风险 2：AI 功能一开始做太深，造成大量返工

策略：

- 先做规则骨架，再让模型做解释
- 所有 AI 输出都保留输入、输出、Prompt 版本、证据片段

### 风险 3：前端先堆页面，后端字段频繁变动

策略：

- 每个模块开始前先冻结最小接口契约
- 先完成 list/create/detail 三类基础接口，再做复杂交互

### 风险 4：当前认证方案不够长期安全

策略：

- V1 先可用
- 在节点 2 评估升级为 refresh token + httpOnly cookie
- 至少补齐 token 过期处理和未授权跳转策略

### 风险 5：文档越写越大，但没人按文档开发

策略：

- 后续以本计划书为执行清单
- `DEV_DOCUMENT.md` 负责目标蓝图
- 每完成一个节点，就补一条“真实完成记录”
