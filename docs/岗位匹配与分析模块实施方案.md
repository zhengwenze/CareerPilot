# CareerPilot 岗位匹配与分析模块实施方案

> 版本：v1.0
> 日期：2026-03-15
> 适用范围：CareerPilot 当前仓库
> 执行原则：先规则闭环，再接 AI 修正；接口、数据结构、页面结构一次定清，后续严格按本方案推进

## 1. 目标定义

本模块要完整覆盖四项核心能力：

1. JD 手动录入与结构化
2. 简历与岗位的智能匹配评分
3. 可解释的匹配报告
4. 规则引擎 + AI 修正的混合评分机制

最终用户闭环：

1. 用户在 `/dashboard/jobs` 录入一个目标岗位 JD
2. 系统自动对 JD 做结构化解析
3. 用户从已有简历中选择一份进行匹配
4. 系统生成总分、维度分、优势、短板、建议、证据
5. 如已配置大模型 API，则在规则结果基础上做 AI 修正与文案增强
6. 匹配报告历史被保存，供后续简历优化、模拟面试、投递追踪复用

## 2. 实施策略

### 2.1 分层原则

- `规则层` 必须独立成立，不依赖模型也能完整出结果
- `AI 层` 只负责修正和增强，不直接替代规则引擎
- `数据层` 以结构化 JSON 为中心，避免后续页面和模型接入频繁改表
- `前端页面` 围绕一个真实工作台完成：JD 列表、JD 编辑、报告生成、报告详情

### 2.2 运行模式

- 未配置模型 API：只跑规则引擎，`model_score = 0`
- 已配置模型 API：规则引擎先产出基线结果，再调用 AI 修正接口
- 模型调用失败：自动降级到规则结果，不影响主流程

## 4. 接口设计

### 4.1 JD 中心

- `POST /jobs`
  - 创建 JD，并立即执行规则结构化
- `GET /jobs`
  - 列表查询，支持 `keyword`、`parse_status`
- `GET /jobs/{job_id}`
  - 获取 JD 详情
- `PUT /jobs/{job_id}`
  - 更新 JD；若 `jd_text` 有变化则自动重新结构化
- `DELETE /jobs/{job_id}`
  - 删除 JD；若已有匹配报告返回 `409`
- `POST /jobs/{job_id}/parse`
  - 手动重试 JD 结构化

### 4.2 匹配报告

- `POST /jobs/{job_id}/match-reports`
  - 请求体：

```json
{
  "resume_id": "uuid",
  "force_refresh": false
}
```

  - 行为：
    - 校验简历已解析成功
    - 校验 JD 已结构化；若未结构化则先补结构化
    - 先跑规则评分
    - 再按配置执行 AI 修正
    - 保存报告并返回

- `GET /jobs/{job_id}/match-reports`
  - 获取某个 JD 的历史匹配报告
- `GET /match-reports/{report_id}`
  - 获取报告详情
- `DELETE /match-reports/{report_id}`
  - 删除报告

## 5. 评分维度

总分 100，采用固定权重：

- `required_skills`：35
- `preferred_skills`：15
- `responsibility_keywords`：20
- `experience_level`：15
- `education_and_location`：15

### 5.1 维度说明

- `required_skills`
  - 比较 JD 必备技能与简历技能、项目、工作经历中的证据覆盖率
- `preferred_skills`
  - 作为加分项，不作为硬性淘汰
- `responsibility_keywords`
  - 比较 JD 职责关键词和简历经历关键词的语义重合度
- `experience_level`
  - 从 JD 经验年限、岗位等级词与简历经历强度做匹配
- `education_and_location`
  - 比较学历要求与地点偏好；若 JD 未写要求则默认不扣分

## 6. 规则引擎设计

### 6.1 JD 结构化引擎

输入：`title/company/job_city/employment_type/jd_text`

输出：

- `basic`
- `required_skills`
- `preferred_skills`
- `required_keywords`
- `education`
- `experience_min_years`
- `responsibilities`
- `benefits`
- `raw_summary`

规则来源：

- 技能词典
- 学历关键词词典
- 年限正则
- 职责/加分项/福利分段规则
- 业务关键词库

### 6.2 匹配引擎

输入：

- `resume.structured_json`
- `resume.raw_text`
- `job.structured_json`
- `job.jd_text`
- `user.profile`（仅作为辅助上下文）

输出：

- `rule_score`
- `dimension_scores_json`
- `gap_json`
- `evidence_json`

处理顺序：

1. 归一化简历与 JD 关键词
2. 建立技能、关键词、职责证据索引
3. 按权重计算各维度分数
4. 生成优势/短板/建议
5. 形成规则基线结果

## 7. AI 修正接口

### 7.1 目标

AI 不直接重做整套评分，只做：

- 修正边界案例
- 生成更自然的解释
- 给出小幅度分数修正
- 保留修正理由和置信度

### 7.2 Provider 抽象

后端统一提供 `AIMatchCorrectionProvider` 接口，支持：

- `disabled`
- `openai_compatible`

后续接入：

- 千问兼容接口
- OpenAI / GPT-5.4
- 其他兼容 Chat Completions 的服务

### 7.3 AI 输入

- 规则评分结果
- 简历结构化数据
- JD 结构化数据
- 规则证据摘要

### 7.4 AI 输出

```json
{
  "score_delta": 4,
  "reasoning": "候选人在项目经历中体现了实验设计，只是规则未完全识别。",
  "strengths": [],
  "gaps": [],
  "actions": [],
  "confidence": 0.74
}
```

约束：

- `score_delta` 必须限制在 `-10 ~ +10`
- AI 修正后总分仍限制在 `0 ~ 100`
- AI 出错时不得阻断主链路

## 8. 前端页面设计

页面：`/dashboard/jobs`

### 8.1 页面结构

- 顶部说明区
  - 模块价值、当前路由、当前状态
- 左列：JD 列表
  - 创建按钮
  - 关键词筛选
  - 解析状态标签
- 中列：JD 编辑区
  - 标题、公司、城市、来源、JD 原文
  - 结构化结果预览
  - 保存、重试结构化、删除
- 右列：匹配区
  - 选择简历
  - 生成报告
  - 历史报告列表
  - 报告详情

### 8.2 交互状态

- 首屏 loading
- 空态：无 JD / 无报告 / 无简历
- error + retry
- 报告生成中按钮禁用
