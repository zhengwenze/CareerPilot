# CareerPilot 岗位匹配与分析模块说明文档

基于《岗位匹配模块开发清单》《岗位匹配与分析模块实施方案》与“简历解析与管理 / 简历中心模块说明文档”整合

> **模块定位**  
> “岗位匹配与分析模块”不是一个单纯的 JD 录入页，也不是一个只返回分数的评分页，而是 CareerPilot 的**岗位目标管理与匹配判断中枢**。它的核心价值，是把目标岗位 JD 转成可保存、可结构化、可解释、可复用的岗位目标数据，并与用户已有简历形成稳定的匹配判断结果，持续供给简历优化、模拟面试、投递追踪等下游模块。

## 一、文档目的

本文档用于统一岗位匹配模块的产品定位、实施边界、数据设计、接口设计、评分规则、页面结构与开发顺序，并结合已经整合完成的“简历解析与管理 / 简历中心模块”口径，明确两者之间的上下游关系。

文档重点解决 4 个问题：

1. 这个模块到底是什么。
2. 第一版必须完成哪些能力，哪些能力暂不纳入本轮范围。
3. 匹配报告应该如何结构化、如何解释、如何保存。
4. 该模块如何承接简历中心，并继续向简历优化、模拟面试、投递追踪供数。

## 二、模块定位与边界

从产品定位上看，岗位匹配模块的本质不是“新建一个页面”，而是建立 CareerPilot 的**岗位目标层与判断层**。其核心闭环为：**录入 JD -> 结构化 -> 选择简历 -> 生成匹配报告 -> 保存历史 -> 向下游复用**。

- **录入 JD：** 用户在 `/dashboard/jobs` 手动录入目标岗位信息与 JD 原文。
- **结构化：** 系统将 JD 文本转换成可匹配的 `structured_json`。
- **选择简历：** 用户从简历中心已有的结构化简历中选取一份作为匹配输入。
- **生成报告：** 系统输出总分、维度分、优势、短板、建议与证据，而不是只给出黑箱分数。
- **保存历史：** 每一次简历与 JD 的组合匹配都形成独立报告，支持回看与后续复用。
- **向下游复用：** 匹配结果将持续供给简历优化、模拟面试、投递追踪等模块。

因此，这个模块更准确的角色是 CareerPilot 的“岗位目标管理与匹配判断中枢”，而不是一个孤立的评分功能。

## 三、模块目标与当前边界

### 3.1 本模块要解决的问题

在当前项目已经具备登录注册、个人资料、求职偏好、简历上传解析与结构化展示之后，产品还缺少真正体现用户价值的“目标岗位上下文”。具体表现为：

1. 用户没有地方长期保存多个目标岗位 JD。
2. 系统无法判断“这份简历是否适合这个岗位”。
3. 后续简历优化、模拟面试、投递追踪都缺少明确的目标岗位输入。

因此，岗位匹配模块的核心目标是：

1. 让用户能保存多个目标岗位 JD。
2. 让用户能从已有简历中选择一份，与某个 JD 做匹配。
3. 让系统输出可解释、可保存、可复用的匹配报告。
4. 在规则引擎稳定成立的前提下，为后续 AI 修正与解释增强预留扩展口。

### 3.2 本次必须完成的能力

第一版必须完成以下闭环能力：

- JD 手动录入
- JD 列表、详情、编辑、删除
- JD 结构化提取
- 选择简历生成匹配报告
- 报告展示总分、维度分、优势项、短板项、行动建议、证据摘要
- 历史匹配报告保存与查看
- 未配置模型 API 时规则闭环可独立成立
- 已配置模型 API 时支持在规则结果基础上做有限 AI 修正与解释增强

### 3.3 本次暂不纳入的能力

第一版先不做：

- 从招聘网站自动抓取 JD
- AI 自动推荐岗位
- 向量检索 / pgvector 参与匹配
- 多份简历批量对比同一个 JD
- 一键跳转生成简历优化建议
- 由 LLM 直接替代规则引擎输出最终分数

这意味着本轮重点不是“做复杂 AI”，而是先把**规则化、可解释、可保存的匹配闭环**跑通。

## 四、与简历中心模块的关系

岗位匹配模块并不是独立起飞，而是直接建立在“简历解析与管理 / 简历中心模块”之上。

### 4.1 上游输入

岗位匹配模块直接消费简历中心沉淀下来的结构化简历资产，重点使用：

- `resumes.structured_json`
- `resumes.raw_text`
- 解析成功或人工修正后的最新版本数据

简历侧重点字段包括：

- `basic_info.summary`
- `education`
- `work_experience`
- `projects`
- `skills.technical`
- `skills.tools`
- `skills.languages`
- `certifications`

### 4.2 本模块新增的中间层能力

如果说简历中心负责“把简历变成结构化资产”，那么岗位匹配模块负责“把岗位目标也变成结构化资产，并产出判断结果”。

也就是说：

- **简历中心**沉淀“输入侧资产”
- **岗位匹配模块**沉淀“目标侧资产 + 判断侧结果”

### 4.3 下游复用方向

岗位匹配模块完成后，最自然的下游扩展包括：

1. 基于 `match_reports` 生成简历优化建议。
2. 为模拟面试提供“岗位要求 + 匹配短板”。
3. 为投递追踪提供“目标 JD + 当前匹配状态”。

因此，该模块是 CareerPilot 从“有简历数据”走向“有岗位判断能力”的关键中枢。

## 五、核心业务闭环

围绕“录入 JD -> 结构化 -> 选择简历 -> 生成报告 -> 保存历史 -> 复用”这一主链路，模块职责可拆成以下 6 个阶段：

### 5.1 JD 录入与保存

- 支持用户手动录入岗位标题、公司、城市、用工类型、来源平台、来源链接与 JD 原文。
- 支持保存多个目标岗位。
- 支持列表查询、详情查看、编辑与删除。

### 5.2 JD 结构化

- 系统在创建或更新 JD 后，对 `jd_text` 执行第一版结构化提取。
- 第一版重点抽取：
  - `basic`
  - `requirements.required_skills`
  - `requirements.preferred_skills`
  - `requirements.required_keywords`
  - `requirements.education`
  - `requirements.experience_min_years`
  - `responsibilities`
  - `benefits`
  - `raw_summary`
- 支持记录 `parse_status` 与 `parse_error`。
- 支持手动重试结构化。

### 5.3 简历选择与匹配触发

- 用户从已有简历中选取一份已解析成功的简历。
- 系统校验简历是否可用。
- 若 JD 尚未完成结构化，可先自动补结构化再进入匹配流程。

### 5.4 匹配评分与报告生成

- 先由规则引擎独立输出基线结果。
- 若已配置模型 API，则在规则结果基础上做小幅 AI 修正与解释增强。
- 输出总分、维度分、优势、短板、建议、证据。
- 匹配结果必须可解释，而不是只给单一数字。

### 5.5 历史记录保存

- 每次“某份简历 + 某个 JD”的匹配结果形成一份独立报告。
- 用户可在同一个 JD 下查看历史报告。
- 报告可删除，但删除 JD 时需考虑已有报告的引用关系。

### 5.6 向下游模块供数

- 匹配报告的分数、短板、建议、证据将成为后续模块的统一输入。
- 本模块形成的数据中心对象包括：`job_descriptions` 与 `match_reports`。

## 六、核心功能总览

| 层次 | 核心功能 | 用户价值 | 当前定位 |
| --- | --- | --- | --- |
| 1 | JD 资产管理 | 保存、查看、编辑、删除多个目标岗位 | 第一版必须完成 |
| 2 | JD 结构化解析 | 将 JD 转为可匹配字段 | 第一版必须完成 |
| 3 | 规则匹配评分 | 不依赖 AI 也能稳定出结果 | 第一版必须完成 |
| 4 | 可解释匹配报告 | 输出总分、维度分、优势、短板、建议、证据 | 第一版必须完成 |
| 5 | AI 修正与解释增强 | 在规则结果上做边界修正与文案增强 | 第一版可接入，必须可降级 |
| 6 | 历史报告沉淀 | 支持回看、复盘与下游复用 | 第一版必须完成 |

## 七、运行模式与实施原则

### 7.1 分层原则

- **规则层**必须独立成立，不依赖模型也能输出完整报告。
- **AI 层**只做修正和增强，不直接替代规则引擎。
- **数据层**以结构化 JSON 为中心，避免后续接 AI 或前端页面时频繁改表。
- **页面层**围绕一个真实工作台完成业务闭环。

### 7.2 运行模式

- **未配置模型 API：** 仅运行规则引擎，`model_score = 0`。
- **已配置模型 API：** 规则引擎先产出基线结果，再调用 AI 修正接口。
- **模型调用失败：** 自动降级到规则结果，不影响主流程。

### 7.3 开发原则

1. 先跑通闭环，再做增强。
2. 先保证可解释，再追求“更智能”。
3. 先让数据结构稳定，再扩展页面和模型能力。

## 八、数据设计

## 8.1 `job_descriptions` 表

用途：保存用户录入的 JD 原文与结构化结果。

建议字段如下：

| 字段名 | 类型 | 说明 | 备注 |
| --- | --- | --- | --- |
| `id` | UUID | JD 主键 | 主键 |
| `user_id` | UUID | 所属用户 | FK -> `users.id` |
| `title` | VARCHAR(255) | 岗位标题 | 必填 |
| `company` | VARCHAR(255) | 公司名称 | 选填 |
| `job_city` | VARCHAR(120) | 岗位城市 | 选填 |
| `employment_type` | VARCHAR(50) | 用工类型 | 选填 |
| `source_name` | VARCHAR(80) | 来源平台 | 选填 |
| `source_url` | TEXT | 原始链接 | 选填 |
| `jd_text` | TEXT | 原始 JD 文本 | 必填 |
| `parse_status` | VARCHAR(20) | 结构化状态 | `pending / success / failed` |
| `parse_error` | TEXT | 结构化失败原因 | 可空 |
| `structured_json` | JSON | 结构化结果 | 可空 |
| `created_at` | TIMESTAMP | 创建时间 | 复用当前 mixin 风格 |
| `updated_at` | TIMESTAMP | 更新时间 | 复用当前 mixin 风格 |
| `created_by` | UUID | 创建人 | 复用当前 mixin 风格 |
| `updated_by` | UUID | 更新人 | 复用当前 mixin 风格 |

建议索引：

- `idx_job_descriptions_user_id`
- `idx_job_descriptions_parse_status`
- `idx_job_descriptions_created_at`

推荐 `structured_json` 结构如下：

```json
{
  "basic": {
    "title": "高级数据分析师",
    "company": "某互联网公司",
    "job_city": "上海",
    "employment_type": "全职"
  },
  "requirements": {
    "required_skills": ["Python", "SQL", "A/B Testing"],
    "preferred_skills": ["Tableau", "Airflow"],
    "required_keywords": ["指标体系", "增长分析", "实验分析"],
    "education": "本科及以上",
    "experience_min_years": 3
  },
  "responsibilities": [
    "负责指标体系建设",
    "支持实验分析与复盘",
    "与产品和运营协作推进增长项目"
  ],
  "benefits": ["双休", "弹性办公"],
  "raw_summary": "偏增长分析方向，强调 Python、SQL、实验分析和跨团队协作"
}
```

说明：

- 第一版不要追求极复杂的岗位知识图谱，重点是能稳定抽出“技能、关键词、经验、学历、城市”等核心字段。
- 后续接 AI 时尽量沿用这套结构，避免前后端重复改造。

## 8.2 `match_reports` 表

用途：保存某一份简历与某一个 JD 的匹配结果。

建议字段如下：

| 字段名 | 类型 | 说明 | 备注 |
| --- | --- | --- | --- |
| `id` | UUID | 报告主键 | 主键 |
| `user_id` | UUID | 所属用户 | FK -> `users.id` |
| `resume_id` | UUID | 参与匹配的简历 | FK -> `resumes.id` |
| `jd_id` | UUID | 目标 JD | FK -> `job_descriptions.id` |
| `status` | VARCHAR(20) | 报告生成状态 | `pending / success / failed` |
| `overall_score` | NUMERIC(5,2) | 综合分 | 0-100 |
| `rule_score` | NUMERIC(5,2) | 规则分 | 0-100 |
| `model_score` | NUMERIC(5,2) | 模型修正分 | 第一版允许为 0 |
| `dimension_scores_json` | JSON | 各维度分数 | 必填 |
| `gap_json` | JSON | 优势、短板、建议 | 必填 |
| `evidence_json` | JSON | 匹配证据 | 必填 |
| `error_message` | TEXT | 失败原因 | 可空 |
| `created_at` | TIMESTAMP | 创建时间 |  |
| `updated_at` | TIMESTAMP | 更新时间 |  |
| `created_by` | UUID | 创建人 |  |
| `updated_by` | UUID | 更新人 |  |

建议索引：

- `idx_match_reports_user_id`
- `idx_match_reports_jd_id`
- `idx_match_reports_resume_id`
- `idx_match_reports_status`
- `idx_match_reports_resume_jd`

推荐 `dimension_scores_json` 结构：

```json
{
  "required_skills": 35,
  "preferred_skills": 12,
  "responsibility_keywords": 18,
  "experience_level": 14,
  "education_and_location": 12,

```

```json
{

    {

```

## 九、接口设计

接口命名应尽量贴合当前仓库风格，优先采用 REST 风格，并继续复用统一响应格式。

## 9.1 JD 中心接口

### 1. 创建 JD

- 方法：`POST /jobs`
- 作用：保存 JD，并触发第一版结构化

请求体建议：

```json
{
  "title": "高级数据分析师",
  "company": "某互联网公司",
  "job_city": "上海",
  "employment_type": "全职",
  "source_name": "Boss直聘",
  "source_url": "https://example.com/job/123",
  "jd_text": "这里是完整 JD 文本"
}
```

### 2. 获取 JD 列表

- 方法：`GET /jobs`
- 作用：用于左侧列表展示
- 建议支持查询参数：`keyword`、`parse_status`

### 3. 获取 JD 详情

- 方法：`GET /jobs/{job_id}`
- 作用：获取原文、结构化结果、基础信息

### 4. 更新 JD

- 方法：`PUT /jobs/{job_id}`
- 作用：编辑标题、公司、JD 文本
- 约定：如果 `jd_text` 变化，自动重置 `parse_status` 并重新解析

### 5. 删除 JD

- 方法：`DELETE /jobs/{job_id}`
- 作用：删除 JD
- 第一版推荐策略：若 JD 已有关联报告，返回 `409 CONFLICT`
- 前端提示：请先删除相关报告或保留该 JD

### 6. 重试 JD 结构化

- 方法：`POST /jobs/{job_id}/parse`
- 作用：当结构化失败后允许手动重试

## 9.2 匹配报告接口

### 7. 创建匹配报告

- 方法：`POST /jobs/{job_id}/match-reports`
- 作用：选择某份简历，为当前 JD 生成一份匹配报告

请求体建议：

```json
{
  "resume_id": "uuid",
  "force_refresh": false
}
```

建议行为：

1. 校验简历已解析成功。
2. 校验 JD 已结构化；若未结构化则先补结构化。
3. 先跑规则评分。
4. 再按配置执行 AI 修正。
5. 保存报告并返回。

### 8. 获取某个 JD 下的匹配报告列表

- 方法：`GET /jobs/{job_id}/match-reports`
- 作用：查看同一个 JD 的历史报告

### 9. 获取匹配报告详情

- 方法：`GET /match-reports/{report_id}`
- 作用：报告详情展示

### 10. 删除匹配报告

- 方法：`DELETE /match-reports/{report_id}`
- 作用：清理历史报告

## 十、规则引擎与评分设计

第一版必须坚持两个原则：

1. 分数必须能解释。
2. 先规则化，再考虑 AI 修正。

## 10.1 总分构成

综合两份方案后，第一版总分采用固定权重 100 分制：

- `required_skills`：35
- `preferred_skills`：15
- `responsibility_keywords`：20
- `experience_level`：15
- `education_and_location`：15

可采用如下公式：

```text
overall_score = required_skills
              + preferred_skills
              + responsibility_keywords
              + experience_level
              + education_and_location
              + ai_correction_delta
```

其中：

- `rule_score` 为规则引擎基础分
- `model_score` 为 AI 修正贡献值，可理解为 `ai_correction_delta`
- 最终总分需限制在 `0 ~ 100`

## 10.2 字段来源

### 简历侧数据来源

使用当前已有的 `resumes.structured_json`，重点读取：

- `basic_info.summary`
- `work_experience`
- `projects`
- `skills.technical`
- `skills.tools`
- `skills.languages`
- `certifications`
- `education`

### JD 侧数据来源

使用 `job_descriptions.structured_json`，重点读取：

- `requirements.required_skills`
- `requirements.preferred_skills`
- `requirements.required_keywords`
- `requirements.education`
- `requirements.experience_min_years`
- `basic.job_city`
- `responsibilities`

## 10.3 各维度规则建议

### A. 必备技能覆盖（35 分）

规则：

```text
命中数 / 必备技能总数 * 35
```

命中策略：

- 大小写归一化
- 去除空格和常见符号差异
- 第一版可加入少量同义词表，如 `js -> javascript`
- 证据来源不仅限于技能列表，也包括项目与工作经历

### B. 加分技能覆盖（15 分）

规则：

```text
命中数 / 加分技能总数 * 15
```

若 JD 没有加分技能：

- 该维度直接记满分 15

这样可以避免因 JD 描述粗糙而让用户被动丢分。

### C. 岗位职责 / 关键词命中（20 分）

规则：

- 将 `required_keywords + responsibilities` 做关键词抽取与去重
- 在简历摘要、工作经历、项目经历中进行命中统计
- 按命中比例映射到 0~20 分

参考公式：

```text
命中关键词数 / 关键词总数 * 20
```

说明：

- 第一版不必引入复杂 NLP，关键词命中足以支撑 MVP。
- 若后续接 AI，可在边界案例中补足规则未识别的语义重合。

### D. 经验级别匹配（15 分）

规则建议：

```text
if 无法识别:
  7
elif 简历年限 >= JD要求:
  15
else:
  max(0, 简历年限 / JD要求 * 15)
```

说明：

- 若无法识别简历年限，应给中间默认分，并在报告中提示“经验信息证据不足”。
- 若简历经验明显满足要求，应记满分。

### E. 学历 / 城市 / 语言等基础条件（15 分）

建议拆成轻量子项，例如：

- 学历匹配
- 城市匹配
- 语言 / 证书 / 其他轻条件

推荐策略：

- 明确命中：给满
- 明确不符：给 0
- 简历未体现：给部分分，并在短板项提示“证据不足”
- 若 JD 未明确写要求，则默认不扣分

## 10.4 报告文案规则

报告至少输出 4 块内容：

1. 总体结论
2. 优势项
3. 短板项
4. 下一步动作

### 总体结论模板

```text
这份简历与目标岗位的匹配度为 {score} 分，整体属于 {high/good/medium/low} 匹配。
当前优势集中在 {top strengths}，主要短板集中在 {top gaps}。
```

### 匹配档位建议

- `85-100`：高匹配
- `70-84`：较匹配
- `50-69`：中等匹配
- `0-49`：低匹配

## 十一、AI 修正设计

## 11.1 AI 的角色

AI 不应直接替代规则引擎，而应在规则结果基础上完成以下任务：

- 修正边界案例
- 生成更自然的解释文案
- 给出小幅度分数修正
- 保留修正理由与置信度

## 11.2 Provider 抽象

后端可统一提供 `AIMatchCorrectionProvider` 接口，支持：

- `disabled`
- `openai_compatible`

后续可扩展到更多兼容 Chat Completions 的服务。

## 11.3 AI 输入与输出

输入建议包含：

- 规则评分结果
- 简历结构化数据
- JD 结构化数据
- 规则证据摘要

输出示例：

```json
{
  "score_delta": 4,
  "reasoning": "候选人在项目经历中体现了实验设计，只是规则未完全识别。",
  "strengths": [],
  "gaps": [],
  "actions": [],
  "confidence": 0.74
}
```

约束：

- `score_delta` 必须限制在 `-10 ~ +10`
- AI 修正后总分仍限制在 `0 ~ 100`
- AI 调用失败时不得阻断主流程，必须自动降级

## 十七、统一后的模块结论

综合《岗位匹配模块开发清单》《岗位匹配与分析模块实施方案》以及已经整合完成的“简历解析与管理 / 简历中心模块说明文档”，可以把本模块统一定义为：

**CareerPilot 的岗位目标管理与匹配判断中枢。**

- 它不是单纯的 JD 录入页。
- 它的核心价值不是“给一个分数”，而是把目标岗位转成可保存、可结构化、可解释、可复用的岗位资产，并产出稳定的匹配判断结果。
- 它的核心业务闭环是：**录入 JD -> 结构化 -> 选择简历 -> 生成匹配报告 -> 保存历史 -> 复用**。
- 它向上承接简历中心输出的结构化简历资产，向下支撑简历优化、模拟面试、投递追踪等后续模块。

## 十八、建议的说明口径

如果后续需要对外汇报、答辩或给团队做模块介绍，可以直接使用下面这段标准表述：

> **标准表述**  
> 岗位匹配与分析模块是 CareerPilot 的岗位目标管理与匹配判断中枢。它负责接收用户录入的目标岗位 JD，完成岗位信息结构化、规则匹配评分、可解释报告生成、历史结果沉淀，并在可选的 AI 修正机制下增强边界案例判断与解释表达，最终将匹配结果持续供给简历优化、模拟面试、投递追踪等后续模块复用。
