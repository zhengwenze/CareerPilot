# CareerPilot 实习面试作战手册

这份文档不是后端知识大全，而是围绕 `CareerPilot` 这个项目整理的一份面试复述手册。

使用方式：

- 先背每题的 `30 秒短答`，保证能快速起势。
- 再熟悉 `2 分钟展开`，保证你能把项目讲出工程感。
- 面试官继续深挖时，优先用 `继续追问` 里的方向接住。
- 如果你发现自己的回答开始飘到项目外，立刻用 `避坑提醒` 收回来。

回答口径统一用第一人称，默认都区分两层：

- `当前项目真实已实现`
- `当前项目没有实现，但如果生产化我会怎么优化`

## 1. 项目整体架构与业务价值

### 1.1 这个项目和直接调用 ChatGPT 有什么区别？

**问题**

这个项目和直接调用 ChatGPT 有什么区别？

**30 秒短答**

我在项目里做的不是一个“套壳问答页”，而是一个连续求职工作流。用户上传主简历后，系统会把 PDF 变成可编辑 Markdown，再沉淀成结构化数据，然后把同一份上下文复用到 JD 分析、定制简历和模拟面试。和直接调用 ChatGPT 最大的区别，是我把状态、版本、权限、存储和失败重试都落到了工程系统里。

**2 分钟展开**

我会先从业务价值讲。直接调用 ChatGPT 的问题是每一步都比较割裂，用户改完简历、换一个 JD、再去做面试时，经常要重复贴上下文，而且上一轮结果不会被系统稳定复用。

我在项目里把这件事拆成了一条连续链路：`resume -> job -> tailored resume -> mock interview`。用户上传 PDF 之后，我先把原件放到 MinIO，把元数据和状态存到 PostgreSQL，再做 PDF 转 Markdown、结构化保存和版本管理。后面用户保存 JD、生成定制简历、进入模拟面试时，系统会知道当前到底基于哪一版 resume、哪一版 JD、哪一份优化结果。

当前项目真实已实现：

- 有登录认证、用户数据隔离、对象存储、数据库持久化和状态字段。
- 简历、JD、定制简历、模拟面试这四条链路是串起来的，不是四个孤立页面。
- AI 返回失败时，我有 timeout、retry、fallback、error category，不是把模型输出原样展示给用户。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会补更强的 workflow orchestration，比如把进程内后台任务升级成独立任务队列。
- 我会补更完整的 observability，比如链路级 tracing、指标面板和报警。
- 我会补更严格的版本回溯能力，明确标识某次面试是基于哪一版定制简历开始的。

**继续追问**

- 如果面试官继续问“那为什么它不是 Prompt Demo”，我会把话题拉到状态管理、文件存储、权限隔离和失败恢复。
- 如果面试官问“为什么这类产品对 ToB 也有价值”，我会强调它把非结构化内容变成可追踪、可复用、可审计的工作流资产。

**避坑提醒**

- 不要只回答“我接了 OpenAI API，所以和 ChatGPT 不一样”，这会显得没有工程深度。
- 不要把项目说成大而全平台，核心点就是 workflow continuity。

### 1.2 你怎么用 1 到 2 分钟介绍这个项目的整体架构？

**问题**

你怎么快速介绍 CareerPilot 的整体架构？

**30 秒短答**

我会说这是一个 `Next.js + FastAPI` 的 AI 求职工作台。前端负责工作台交互，后端负责认证、简历和 JD 工作流、对象存储访问、AI 调用和状态编排。持久化上我用 PostgreSQL 存业务数据，用 MinIO 存简历原件，用 Redis 做 JWT 黑名单，后台异步流程目前用 `asyncio.create_task` 在应用进程内驱动。

**2 分钟展开**

我一般会按“入口层、业务层、存储层、AI 层”来讲。

入口层是 Next.js 工作台，用户从 dashboard 进入 resume、interviews 等页面。后端是 FastAPI，路由层比较薄，主要做参数校验、鉴权和依赖注入；核心逻辑都放在 service 层，比如 resume、job、tailored_resume、mock_interview。

业务层的核心是四个实体和它们的关系：主简历、目标 JD、定制简历会话、模拟面试会话。为了让后续链路知道自己消费的是哪份输入，我在 resume 和 job 上都维护了 `latest_version`，在定制简历和模拟面试会话里记录 `source_resume_version` 和 `source_job_version`。

存储层里，PostgreSQL 存业务主数据和状态，MinIO 存 PDF 原件，Redis 目前主要用于 JWT 登出后的 token blocklist。AI 层我没有做成一个大而泛的 agent 平台，而是按业务域拆开：简历解析增强、定制简历生成、模拟面试问答和点评分别走对应的 prompt 和 provider 配置。

当前项目真实已实现：

- FastAPI + SQLAlchemy + PostgreSQL + Redis + MinIO 的分层。
- 业务状态、错误响应、健康检查、readiness 检查。
- AI provider 的 timeout、retry、fallback 和错误分类。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会把后台任务从进程内调度迁移到专门的 worker。
- 我会把 AI 和业务链路的监控指标拆出来，做更明确的 SLA 观测。
- 我会补更严格的限流、配额和审计能力。

**继续追问**

- 如果继续问“为什么不直接 all in Redis 或 all in database”，我会把话题转到数据类型和访问模式。
- 如果继续问“系统最脆弱的点在哪”，我会直接说是 AI 调用链和进程内异步任务的可靠性。

**避坑提醒**

- 不要把架构图讲成组件清单，要围绕“这条求职链路怎么跑通”来讲。
- 不要吹成微服务架构，当前项目不是。

### 1.3 为什么这个项目对 ToB 企业软件岗位也有解释力？

**问题**

这个项目明明是求职产品，为什么对 ToB 企业软件实习面试也有帮助？

**30 秒短答**

因为它本质上是在处理非结构化文档到结构化工作流资产的转化，这和很多 ToB 软件做合同、工单、知识库、报告流转时遇到的问题很像。我的重点不是“做了一个简历产品”，而是把文档解析、状态编排、权限隔离和失败恢复做成了可持续的业务链路。

**2 分钟展开**

我在项目里做的很多问题，其实是典型 ToB 题型。比如用户上传 PDF 后怎么做原件存储和元数据分离，怎么处理状态流转，怎么保证多步流程不丢上下文，怎么把 AI 的不稳定输出变成业务上可消费的结果，怎么做用户隔离和错误分类。

这类思路可以平移到很多企业软件场景，比如销售跟进助手、知识库问答、标书分析、简历筛选、报告生成。ToB 面试官通常关心的不是你产品的具体题材，而是你有没有把一条复杂链路拆成可维护的系统。

当前项目真实已实现：

- 文档原件、结构化内容和业务状态分层存储。
- 不同步骤之间复用同一份业务上下文，而不是每次重新输入。
- AI 输出不是直接展示，而是先过结构化和错误分类。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会补租户级权限模型，而不只是单用户隔离。
- 我会补操作审计、字段级权限和更细的回放能力。
- 我会补批量任务和管理员视角的工作台。

**继续追问**

- 如果继续问“你觉得哪一点最像企业软件”，我会强调状态管理和可追踪性。
- 如果继续问“AI 在 ToB 场景最大风险是什么”，我会说是输出不稳定和可追责性不足。

**避坑提醒**

- 不要硬说“我的项目就是 ToB”，更好的说法是“它解决的问题和 ToB 系统有共性”。

## 2. 登录认证与用户数据隔离

### 2.1 你这个项目的登录认证是怎么做的？

**问题**

你这个项目的登录认证方案是什么？

**30 秒短答**

我在项目里用的是比较标准的 `JWT access token` 方案。用户注册和登录成功后，后端签发 access token；前端后续请求带 Bearer token；后端通过依赖注入统一解 token 并拿当前用户。这个方案简单直接，适合当前 MVP。

**2 分钟展开**

我在 FastAPI 里把认证放在 `auth` 路由和通用依赖里。注册和登录成功之后，后端会创建一个带 `sub`、`jti`、`exp` 和 `type` 的 access token。后续请求进来时，先从 Authorization header 里提取 Bearer token，再做解码、类型校验和 blocklist 检查，最后根据 `sub` 去拿当前用户。

这样做的好处是后端路由层比较干净，大部分业务路由只需要依赖 `get_current_user`，不用每个接口重复写认证逻辑。对实习项目来说，这个复杂度和收益是比较平衡的。

当前项目真实已实现：

- 注册、登录、登出、`/auth/me`。
- token 内有 `jti` 和过期时间。
- 统一依赖层完成 Bearer token 解析、JWT 解码和当前用户获取。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会补 refresh token 体系，避免 access token 过期后用户体验太差。
- 我会补设备管理、异常登录告警和更细的登录审计。
- 我会把密码策略、验证码和风控加上去。

**继续追问**

- 如果继续问“为什么不用 session”，我会回答当前前后端分离 + API 架构下，JWT 更轻量、部署更灵活。
- 如果继续问“JWT 的问题是什么”，我会引到 token 作废和 blocklist。

**避坑提醒**

- 不要只说“我用了 JWT，很常见”，要讲出字段、流转和当前边界。
- 不要假装项目已经有 refresh token，目前没有。

### 2.2 JWT + Redis Token Blocklist 的作用是什么？

**问题**

JWT + Redis Token Blocklist 的作用是什么？

**30 秒短答**

JWT 本身是无状态的，签发出去之后，单靠后端没法主动让它立刻失效。所以我在项目里加了 Redis blocklist：用户登出时把 token 的 `jti` 按剩余 TTL 写进 Redis，后续每次鉴权都查一次。如果命中 blocklist，就算 JWT 还没过期，也不能再用。

**2 分钟展开**

这是一个很典型的工程补丁。JWT 好处是轻量、易扩展、不依赖服务端 session 表；但缺点是它天然不支持“立刻注销”。如果用户点了退出登录，而 token 还剩几十分钟有效期，那单靠前端删本地 token 是不够的，只要这个 token 被拷走，还能继续调用接口。

我在项目里的做法是：签 token 时写入 `jti`，登出时把 `jti` 写入 Redis，key 形如 `auth:blocklist:{jti}`，TTL 就是 token 剩余过期时间。这样不会永久堆积垃圾数据，同时也解决了“JWT 无法主动失效”的问题。

当前项目真实已实现：

- token payload 里包含 `jti`。
- 登出接口会把 `jti` 写入 Redis blocklist。
- 每次请求鉴权都会检查 blocklist。
- 如果 Redis 不可用，系统会 fallback 到内存 blocklist，但这只适合开发环境或单进程场景。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会补 refresh token + rotate 机制，让注销和续期都更完整。
- 我会补多端登录策略，比如单端退出还是全端退出。
- 我会把认证事件接到审计日志里。

**继续追问**

- 如果继续问“为什么不直接把 token 存数据库”，我会说当前只需要解决注销后的失效问题，Redis 的 TTL 模型更合适。
- 如果继续问“为什么不只靠 JWT 过期时间”，我会强调安全和用户主动退出的语义。

**避坑提醒**

- 不要说 Redis blocklist 是“为了缓存登录态”，它在这个项目里主要不是做缓存，而是做失效控制。
- 不要把内存 blocklist 说成生产方案，它只是兜底。

### 2.3 用户数据隔离你是怎么做的？

**问题**

你怎么保证 A 用户看不到 B 用户的简历、JD 和面试数据？

**30 秒短答**

我在项目里做的是后端强隔离，不依赖前端隐藏按钮。核心做法是所有业务主表都带 `user_id`，查询详情和列表时都按当前用户过滤，拿不到就返回 404。测试里我也专门做了 cross-user case，验证 resume、job 和 tailored resume workflow 都不会串数据。

**2 分钟展开**

这类问题我一定强调“以后端为准”。因为前端就算不显示别人的数据，只要后端接口没做所有权校验，用户改个 ID 一样能撞库。

我在项目里，像 `get_resume_for_user`、`get_job_or_404` 这种服务方法，都会带 `current_user` 参与查询，SQL 条件里明确要求 `resource.user_id == current_user.id`。如果不匹配，后端就返回 404 或 401/403，不会把资源暴露出去。这个思路也贯穿到后续 workflow，比如 tailored resume 和 mock interview 也是先校验关联资源是否属于当前用户。

当前项目真实已实现：

- 主业务表都和 `user_id` 绑定。
- 列表接口和详情接口都按当前用户过滤。
- 测试里覆盖了 A 用户创建资源、B 用户访问应失败的场景。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会补更细粒度的授权模型，比如团队协作、管理员代查、租户隔离。
- 我会补更明确的审计日志，记录谁在什么时候访问了什么资源。
- 我会补资源级别的操作权限矩阵，而不只是 owner-only。

**继续追问**

- 如果继续问“为什么有时候返回 404 而不是 403”，我会说这是为了减少资源存在性泄露。
- 如果继续问“光靠 user_id 够不够”，我会说对当前单租户 MVP 足够，但多租户场景还要引入 tenant 维度。

**避坑提醒**

- 不要回答成“前端拿不到别人的 token，所以安全”，这不是后端视角。
- 不要忘了强调测试验证，不然容易像只停留在理论设计。

## 3. 简历 / JD / 定制简历 / 模拟面试核心业务链路

### 3.1 用户上传简历后，后端完整链路是什么？

**问题**

用户上传简历后，后端完整链路是什么？

**30 秒短答**

我在项目里的链路是：先校验文件类型和大小，再把 PDF 原件上传到 MinIO，同时在 PostgreSQL 里创建 resume 记录和 parse job，接着用 `asyncio.create_task` 异步启动解析任务。解析任务会把 PDF 转成 Markdown，写回 `raw_text` 和 `parse_artifacts_json`，用户保存编辑后再转成 `structured_json`，后续 JD 匹配、定制简历和模拟面试都复用这份结构化结果。

**2 分钟展开**

我会按“上传、落库、异步解析、结构化沉淀、下游复用”五步讲。

第一步是上传。后端先限制只收 PDF，并检查文件大小。第二步是存储分离：PDF 原件写 MinIO，数据库只保存 `file_name`、`content_type`、`file_size`、`storage_bucket`、`storage_object_key` 和下载 URL 所需元数据。

第三步是创建异步解析任务。resume 记录初始是 `parse_status = pending`，同时创建一条 parse job，然后通过 `schedule_resume_parse_job` 用 `asyncio.create_task` 启动后台解析。

第四步是解析。后台任务会更新 parse job 和 resume 状态到 `processing`，然后做 PDF 转 Markdown。如果 AI 成功，就把清洗后的 Markdown 写回；如果 AI 出问题，会根据 timeout、provider error、格式错误等情况记录错误，并在需要时走 fallback。

第五步是结构化沉淀。当前项目里 PDF 解析成功后，核心产物首先是 Markdown。用户在前端编辑并保存 Markdown 时，后端会调用 Markdown parser，把结果转成 `structured_json`，同时 `latest_version + 1`。后续的 tailored resume 和 mock interview 主要消费的是这份结构化数据和对应版本。

当前项目真实已实现：

- 上传时的基础校验。
- MinIO 原件存储 + PostgreSQL 元数据存储。
- parse job + `parse_status` 状态流转。
- Markdown 保存后转结构化 JSON，并标记下游 report stale。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会加防病毒扫描和更强的文件安全检查。
- 我会把后台任务迁移到独立 worker，避免应用重启导致任务丢失。
- 我会补更细粒度的 pipeline 可视化状态，而不只是最终 parse status。

**继续追问**

- 如果继续问“为什么 PDF 解析后不是直接生成 structured_json”，我会说 Markdown 是人可编辑的中间事实层，方便人工修正。
- 如果继续问“为什么要保留原件”，我会说是为了下载、回放、问题定位和后续重解析。

**避坑提醒**

- 不要把“AI 解析成功”说成“数据就一定正确”，这个项目里 Markdown 仍然需要人工可编辑。
- 不要忘了讲 parse job 和状态，否则链路会显得太黑盒。

### 3.2 简历解析状态、JD 解析状态、定制简历生成状态是怎么设计的？

**问题**

简历解析状态、JD 解析状态、定制简历生成状态如何设计？

**30 秒短答**

我在项目里没有用一个“大一统状态机”硬套所有流程，而是按业务对象分别设计。resume 和 job 主要用 `parse_status` 表示解析阶段，定制简历额外有更贴近前端展示的 `display_status`。这样做的原因是不同流程关注点不一样：简历和 JD 更像解析任务，定制简历更像一个多段生成任务。

**2 分钟展开**

resume 和 job 这两类对象的核心问题是“内容有没有解析好”，所以我给它们设计的是 `pending / processing / success / failed` 这类解析状态。resume 还会把更多细节放在 `parse_artifacts_json` 里，比如 `ai_used`、`fallback_used`、`ai_error_category`、`canonical_resume_md` 等，避免主表字段膨胀。

定制简历不一样。它不只是一个解析任务，而是一个带中间进度和用户可感知结果的生成过程。所以我在响应层又做了 `display_status`，包括 `processing`、`segment_progress`、`success`、`failed`、`empty_result` 等。这样前端更容易根据状态决定展示“处理中”“有分段进度”“失败可重试”还是“结果为空”。

另外，我在 resume、job 上都维护了 `latest_version`，在定制简历和模拟面试会话里记录 `source_resume_version` 和 `source_job_version`。这样当上游内容更新时，我能知道当前结果是不是基于旧版本生成的。

当前项目真实已实现：

- resume / job 有 `parse_status`。
- tailored resume 有 `display_status`、`retryable`、`error_message`、`fallback_used` 等面向业务消费的字段。
- 版本字段贯穿上下游。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会把状态事件单独抽成更标准的 event log，便于回放。
- 我会补更统一的 workflow state machine 定义，减少散落在 service 里的状态判断。
- 我会给每条状态转换补指标和告警。

**继续追问**

- 如果继续问“为什么不所有状态都塞主表”，我会说主表只保留高价值字段，详细诊断放 JSON 或任务态里更灵活。
- 如果继续问“状态不同步怎么办”，我会引到幂等、事件记录和后台任务可靠性。

**避坑提醒**

- 不要只背状态值，要解释为什么 resume 和 tailored resume 不共用一套状态。
- 不要假装现在已经有完整 BPMN 式工作流引擎，当前没有。

### 3.3 mock interview 为什么能复用 resume / job / tailored resume 上下文？

**问题**

模拟面试为什么能复用前面的上下文，而不是让用户重新贴一次资料？

**30 秒短答**

因为我在项目里把 mock interview 设计成 workflow 的下游，而不是独立功能。创建会话时，后端会把 resume、job、match report 以及可用的 tailored resume 一起挂到 session 上，同时记录 `source_resume_version` 和 `source_job_version`。这样后续出题和点评都是基于同一份业务上下文继续跑。

**2 分钟展开**

如果把模拟面试做成一个独立页面，用户每次都要重新输入岗位信息和简历摘要，体验很差，而且前后结果也对不齐。所以我在项目里选择让它直接消费上游产物。

具体来说，创建 mock interview session 时，后端会先确认当前 resume markdown 是否 ready、match report 是否 ready，然后根据是否有 tailored resume 决定 `resume_source_kind`。如果定制简历可用，就优先用它；如果没有，也能 fallback 到原始 resume。随后把 `resume_id`、`jd_id`、`match_report_id`、`optimization_session_id`、`source_resume_version`、`source_job_version` 这些信息全部落到 session 上。

这样做的价值是两层。第一层是业务连续性，用户不需要重新整理背景。第二层是可追踪性，我能知道这次面试到底基于哪一版输入材料开始。如果后续用户改了简历，我也能知道旧会话是历史版本。

当前项目真实已实现：

- 会话创建时复用已有 resume、job、match report 和可用的 tailored result。
- 如果 tailored markdown 缺失，可以回退到原始 resume。
- session 里记录 source version。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会在前端显式展示“本次面试基于哪一版简历 / JD”。
- 我会补会话启动时的快照摘要，避免上游对象后续变化影响历史解释。
- 我会加 resume source 切换策略，让用户明确选择“用原简历还是定制简历开始面试”。

**继续追问**

- 如果继续问“为什么还需要 match report”，我会说它是 resume 和 JD 之间的桥梁，能帮助问题更贴岗。
- 如果继续问“如果上游版本更新怎么办”，我会说当前通过 source version 识别，生产化时可进一步做 stale 提示。

**避坑提醒**

- 不要把这个能力描述成“AI 自己记住了上下文”，真正记住的是后端的数据模型和会话绑定关系。

### 3.4 定制简历和模拟面试之间为什么要保持 workflow continuity？

**问题**

为什么你强调定制简历和模拟面试要连起来，而不是拆成两个独立模块？

**30 秒短答**

因为这两个模块服务的是同一个用户任务。用户做定制简历，本质上是在强化岗位相关证据；进入模拟面试后，最应该练的也是这些岗位相关证据。如果把它们拆开，用户就得重新解释背景，系统也无法保证前后口径一致。

**2 分钟展开**

我在项目里不是把“简历优化”和“面试训练”当成两个单点功能，而是当成同一个 workflow 的前后阶段。定制简历阶段会沉淀更贴岗的表达和证据，面试阶段则消费这些结果，进一步测试用户是否真的能把这些经历讲清楚。

从工程上看，这种 continuity 让数据复用更自然：同一个 job、同一个 match report、同一版 resume 都能沿链路往下传。从产品上看，用户不会觉得自己在使用四个割裂工具，而是在完成一件连续任务。

当前项目真实已实现：

- 定制简历结果可以直接 handoff 到 mock interview。
- mock interview 会话创建时会判断可用的 resume source。
- 上下文对象之间已有明确关系字段。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会补 workflow timeline，让用户看到自己当前卡在哪一步。
- 我会补从面试反馈反推回定制简历的闭环，比如把薄弱点反写成下一轮优化任务。

**继续追问**

- 如果继续问“为什么这不是产品层面的事情而是后端事情”，我会说 continuity 最终要靠数据模型和状态链路落地。

**避坑提醒**

- 不要只讲产品故事，面试官更关心你怎么让这个连续性真正成立。

## 4. AI 调用链稳定性：Prompt、结构化输出、超时、重试、降级、错误分类

### 4.1 AI 调用超时、空响应、格式错误、上下文过长时怎么处理？

**问题**

AI 调用超时、空响应、格式错误、上下文过长时怎么处理？

**30 秒短答**

我在项目里把 AI 当成不稳定上游来处理，不会假设它每次都正常。做法是分层超时、有限重试、错误分类、fallback 和结构化校验。超时和可恢复错误会重试；空响应和格式错误会被识别成明确错误类型；必要时走保守 fallback，而不是把坏结果直接返给用户。

**2 分钟展开**

我这里最重要的思路是“AI 不是业务真相，而是一个可能失败的外部依赖”。所以我在 client 层先抽象 provider config，把 timeout、connect timeout、read timeout、retry budget 这些基础能力统一起来。

对超时，我在请求层做了 timeout 配置，在部分链路还会加总预算，避免重试把单次任务拖得无限长。对空响应和格式错误，我会在消费阶段做检查，比如 text completion 返回空字符串就直接当错误，JSON completion 会先抽取 JSON 再解析，如果不是 object 或者 JSON 非法，也会打成 `invalid_response_format` 或 `json_decode_error`。

对上下文过长，当前项目没有做特别复杂的 token 裁剪框架，但我不会把这个问题藏起来。我的回答会是：现在我通过控制 payload 范围、拆业务 prompt、限制 max tokens 来降低风险；如果生产化，我会加更显式的 prompt budget 和内容裁剪策略。

当前项目真实已实现：

- timeout、retry、错误分类。
- 对空响应、非法 JSON、stream 异常都有明确错误处理。
- 某些链路有 fallback，比如简历解析和定制简历生成。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会加 token budget 预估和输入裁剪。
- 我会做 prompt 压缩和长文本分段摘要。
- 我会补 provider 级熔断和限流。

**继续追问**

- 如果继续问“什么错误会重试”，我会回答 timeout、可恢复的 5xx、连接中断这类 transient error。
- 如果继续问“什么错误不该重试”，我会回答明显格式错误、鉴权错误、业务校验错误。

**避坑提醒**

- 不要说“超时就多试几次”，面试官会继续追问重试风暴和总时延。
- 不要假装现在已经有完整 token 计数器，当前没有做到那么重。

### 4.2 如何保证大模型输出结果稳定可用？

**问题**

如何保证大模型输出结果稳定可用？

**30 秒短答**

我在项目里做的不是追求“每次都一样”，而是保证“每次都可消费”。核心方法是缩小 prompt 职责、约束输出格式、做结果校验、识别错误类型、必要时 fallback 到保守结果。只要业务链路还能继续，就算 AI 质量波动，系统也不会直接崩。

**2 分钟展开**

我一般从三层讲这个问题。

第一层是 prompt 分层。我没有用一个超级大 prompt 同时做所有事，而是按业务拆开，比如简历清洗、定制简历、模拟面试问题生成和点评分别处理，这样每次任务更聚焦，输出也更稳。

第二层是结构化消费。像 JSON 输出，我会要求它返回对象，再由后端解析和校验；像定制简历，我最终要的不是一段自由文本，而是结构化 document 和可下载 Markdown；这样即使模型有波动，我也知道哪里失败了。

第三层是兜底。比如定制简历如果 AI 不可用，我会生成保守 fallback 文档，至少保留原始结构和已有事实，不让用户直接拿到一份空白结果。

当前项目真实已实现：

- Prompt 按业务域拆分。
- JSON / Markdown 结果会经过后端结构化解析。
- fallback、错误分类、重试、空结果判断已经在代码里体现。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会补更严格的 schema validation 和自动修复。
- 我会做离线评测集，定期回归 prompt 质量。
- 我会给关键场景加人工 review 开关。

**继续追问**

- 如果继续问“稳定可用和稳定一致是不是一回事”，我会说不是，业务更关心可消费和可恢复。
- 如果继续问“你怎么避免胡编”，我会说靠结构化输入、事实边界和 fallback 策略。

**避坑提醒**

- 不要说“我把 temperature 调低就稳定了”，这太浅。
- 不要把“模型表现好”当成系统稳定性的全部来源。

### 4.3 codex2gpt、Ollama 和 fallback 在你的项目里分别扮演什么角色？

**问题**

codex2gpt、Ollama 和 fallback 在你的项目里分别做什么？

**30 秒短答**

我把它们看成三层：`codex2gpt` 是主力上游，负责主要 AI 能力；`Ollama` 可以作为次级 provider；`fallback` 不是另一个模型，而是当 AI 不可用时的保守业务兜底，保证系统还能返回可解释结果。

**2 分钟展开**

在我的实现里，AI provider 是可配置的。比如简历解析增强默认可以走 `codex2gpt`，也可以配置 secondary provider，比如本地 `Ollama`。如果主 provider 超时或者输出异常，系统会根据链路不同决定要不要切 secondary，或者直接走规则 / 保守结果。

这个设计的重点不是“多接几个模型”，而是把模型当成可替换依赖。因为对业务系统来说，最怕的不是某个模型效果波动，而是整个流程因为单一上游挂掉直接不可用。

当前项目真实已实现：

- provider 配置抽象。
- primary / secondary 的思路已经落到简历解析链路里。
- 定制简历存在保守 fallback 文档。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会补 provider 级路由策略，比如按场景、成本或稳定性切流。
- 我会补更标准的熔断、回退和 provider 健康度管理。
- 我会补自动 A/B 评测，而不是纯手工换 provider。

**继续追问**

- 如果继续问“fallback 会不会把错误掩盖掉”，我会说不会，fallback 同时会记录 warning 和错误信息。
- 如果继续问“为什么不用多个 provider 并发投票”，我会说当前 MVP 不值得引入这么高的成本和复杂度。

**避坑提醒**

- 不要把 fallback 说成“另一个更差的模型”，它本质上是业务兜底策略。
- 不要吹成智能路由平台，当前没有做到那一步。

## 5. 存储设计：PostgreSQL + Redis + MinIO

### 5.1 为什么用 MinIO 存 PDF，而不是直接存数据库？

**问题**

为什么用 MinIO 存 PDF，而不是直接存数据库？

**30 秒短答**

因为 PDF 原件是大对象，适合放对象存储，不适合塞进关系型数据库。我的做法是把原件放 MinIO，把文件元数据和业务状态放 PostgreSQL。这样数据库更适合做查询和事务，对象存储更适合做大文件保存和下载。

**2 分钟展开**

我在项目里做的是典型的“元数据和二进制分离”。如果把 PDF 直接存 PostgreSQL，短期是方便，但长期会带来几个问题：数据库体积增长快、备份恢复更重、查询和存储职责混在一起、文件下载也不自然。

所以我选择让 MinIO 保存 PDF 原件，数据库只记录 `storage_bucket`、`storage_object_key`、`file_name`、`content_type`、`file_size` 等元数据。这样一来，业务查询主要打数据库；文件读写和下载走对象存储；需要给前端下载时，我再生成 presigned URL。

当前项目真实已实现：

- PDF 原件存 MinIO。
- PostgreSQL 保存元数据和对象 key。
- 下载通过 presigned URL 返回，而不是后端自己搬运整个文件流。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会加对象存储生命周期策略和冷热分层。
- 我会加文件校验摘要，比如 hash，方便去重和篡改检测。
- 我会加更严格的上传安全扫描。

**继续追问**

- 如果继续问“什么时候可以把文件放数据库”，我会说小附件、低规模内部系统有时可以，但这个项目不合适。
- 如果继续问“MinIO 和 OSS/S3 的关系”，我会说 MinIO 是兼容 S3 的对象存储方案，适合本地和私有部署。

**避坑提醒**

- 不要回答成“因为大家都这么做”，要讲清楚大对象和元数据的职责边界。
- 不要说现在已经做了对象生命周期管理，当前没有。

### 5.2 为什么用 Redis？Redis 在项目里具体解决了什么问题？

**问题**

为什么用 Redis？Redis 在项目里具体解决了什么问题？

**30 秒短答**

我在这个项目里用 Redis 主要不是为了炫技做缓存，而是为了解决 `JWT 无法主动失效` 的问题。用户登出后，我把 token 的 `jti` 写进 Redis blocklist，并带上 TTL，这样后端后续就能拒绝一个虽然没过期、但已经被用户主动注销的 token。

**2 分钟展开**

我会非常诚实地回答这个问题：当前项目里 Redis 的核心落地点其实比较克制。它最明确解决的是认证上的 blocklist 问题，而不是承担主业务状态存储。原因是项目现在还是 MVP，我不想为了显得系统复杂，就把状态和缓存到处塞进 Redis。

Redis 适合这个场景，是因为 token 失效本身就是一个天然带 TTL 的临时状态。把 `jti` 放进去，等 token 过期后这条记录自然淘汰，模型很干净。

当前项目真实已实现：

- Redis blocklist。
- readiness 检查时会验证 Redis 可用性。
- 如果 Redis 配置不存在，会退回内存 blocklist，便于本地开发。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会把更多适合短期保存的会话级状态或限流状态放到 Redis。
- 我会补分布式限流和热点保护。
- 我会评估是否需要缓存某些高频只读结果，但前提是有真实访问瓶颈。

**继续追问**

- 如果继续问“为什么不把解析状态放 Redis”，我会说当前状态要持久可追踪，所以核心状态仍然放 PostgreSQL。
- 如果继续问“Redis 宕机会怎样”，我会说当前会影响 blocklist 准确性，所以生产化要把 Redis 当正式依赖来监控和兜底。

**避坑提醒**

- 不要把项目吹成重缓存架构，当前 Redis 使用很克制。
- 不要为了显得高级胡扯缓存击穿和大 Key，这个项目里没有那个复杂度。

### 5.3 PostgreSQL、Redis、MinIO 在项目里分别存什么？

**问题**

PostgreSQL、Redis、MinIO 在项目里分别存什么？

**30 秒短答**

我在项目里是按数据性质拆的：PostgreSQL 存业务主数据和状态，Redis 存短生命周期的 token 黑名单，MinIO 存简历 PDF 原件。这样每种存储都做自己最擅长的事，不会把关系查询、大对象存储和临时状态混在一起。

**2 分钟展开**

PostgreSQL 里存的是业务事实，比如用户、resume、job、match report、resume optimization session、mock interview session 和 turn。这些数据要支持查询、关联、版本控制和状态回溯，所以适合放关系库。像 `structured_json`、`diagnosis_json` 这类半结构化数据，我会放 JSON 字段，避免为低复用字段设计太多表。

Redis 现在主要存认证相关的临时状态，也就是 blocklist。它不是这个项目的 source of truth。

MinIO 存的是 PDF 原件，因为它们是大对象，访问模式和数据库里的业务字段完全不同。

当前项目真实已实现：

- PostgreSQL 承担主业务数据和状态。
- Redis 只承担短时认证状态。
- MinIO 承担文件原件。
- 下载通过数据库元数据找到对象存储 key，再返回 presigned URL。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会给 PostgreSQL 做更细的归档和表增长治理。
- 我会给 MinIO 对象增加 hash、审计和生命周期策略。
- 我会给 Redis 补高可用部署和更清晰的依赖降级策略。

**继续追问**

- 如果继续问“为什么 structured_json 不拆更多表”，我会说当前阶段优先追求迭代速度和 schema 演化灵活性。
- 如果继续问“JSON 字段会不会难查”，我会说高频查询字段仍放显式列，JSON 主要承载半结构化内容。

**避坑提醒**

- 不要说“Redis 也是数据库，所以都差不多”，三者职责完全不同。
- 不要把 JSON 字段说成万能方案，它只是当前阶段的平衡。

### 5.4 版本字段、状态字段、JSON 字段和索引你是怎么考虑的？

**问题**

为什么你会在这些表里设计版本字段、状态字段、JSON 字段和索引？

**30 秒短答**

因为这个项目不是单次请求就结束，而是多步 workflow。版本字段解决“当前结果基于哪一版输入”的问题，状态字段解决“当前流程跑到哪一步”的问题，JSON 字段解决“半结构化结果迭代快”的问题，索引则保证按用户、状态、时间这些常见维度查询时不会太慢。

**2 分钟展开**

我在 resume 和 job 上都有 `latest_version`，因为用户可能先上传，再编辑 Markdown，再修改 JD。如果没有版本字段，下游的 tailored resume 和 mock interview 就不知道自己消费的是哪一版数据。

状态字段是为了让 workflow 可观测。像 `parse_status`、`display_status`、`status_stage` 这些字段，都是为了让后端和前端都能知道当前任务在什么阶段、能不能继续下一步。

JSON 字段则是工程上的平衡。比如 `structured_json`、`tailored_resume_json`、`review_json` 这些内容结构复杂，而且迭代期变化快。如果一开始就完全范式化，开发成本会很高；但我也没有把高频查询字段全都塞 JSON，高频字段还是有显式列。

索引上，我优先给 `user_id`、`status`、`created_at` 和会被高频关联的外键加索引，因为这些是列表筛选和详情查询最常用的维度。

当前项目真实已实现：

- `latest_version` 和 `source_*_version`。
- 多类状态字段。
- 多个 JSON 字段承载半结构化结果。
- 按用户、状态、时间、外键的基础索引。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会结合真实 SQL 和执行计划继续调索引。
- 我会对高频 JSON 查询场景做专门优化，必要时拆表。
- 我会引入更明确的数据归档策略。

**继续追问**

- 如果继续问“为什么不一开始就强范式化”，我会说当前迭代速度比 schema 完美更重要。
- 如果继续问“索引会不会影响写入”，我会说会，所以只加高价值索引。

**避坑提醒**

- 不要说“加索引就是性能优化”，要先说访问模式。
- 不要把 JSON 字段包装成最佳实践，它是阶段性取舍。

## 6. 项目缺陷、故障排查与生产级优化

### 6.1 `asyncio.create_task` 做后台任务有什么优点和风险？

**问题**

`asyncio.create_task` 做后台任务有什么优点和风险？

**30 秒短答**

我在当前项目里用 `asyncio.create_task` 驱动简历解析、JD 解析、定制简历和模拟面试的后台流程，优点是实现非常轻、链路短、开发效率高；风险是任务可靠性绑定在应用进程上，进程重启、扩容和多实例调度时会比较脆弱，所以它适合 MVP，不适合直接当生产最终方案。

**2 分钟展开**

我选择它的原因很现实：当前项目阶段更看重把 workflow 跑通，而不是一开始引入复杂任务系统。用 `create_task` 可以在用户请求结束后继续跑耗时操作，同时不用把整个服务拆开。

但我也很清楚它的边界。第一，任务生命周期跟应用进程绑定，服务重启时可能中断；第二，多实例部署时，任务状态不天然共享；第三，可靠重试、死信、调度可视化这些能力都比较弱；第四，如果任务里有阻塞操作或者异常处理不当，也会影响进程稳定性。

当前项目真实已实现：

- 用 `create_task` 驱动多个后台流程。
- 通过 active task id 做了基础去重。
- 对部分任务失败会记录日志和错误状态。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会把任务迁移到独立 worker + 持久化任务队列。
- 我会补更标准的幂等 key、重试策略和失败补偿。
- 我会给任务系统补监控、报警和管理面板。

**继续追问**

- 如果继续问“为什么现在不用 Celery 或更重的队列”，我会说 MVP 阶段先控制复杂度。
- 如果继续问“那你怎么保证用户知道任务没完成”，我会引到状态字段和前端轮询。

**避坑提醒**

- 不要把 `create_task` 说成生产级可靠任务框架。
- 不要只讲优点不讲风险，面试官通常就是在等你自己承认边界。

### 6.2 当前项目最大的问题是什么？如果生产化，你会先重构哪里？

**问题**

当前项目最大的问题是什么？如果生产化，你会先重构哪里？

**30 秒短答**

我认为当前项目最大的问题是“核心 workflow 已经成立，但后台任务和 AI 调用链的可靠性还不够生产级”。如果要先重构一处，我会先从后台任务体系下手，因为它直接影响简历解析、定制简历和模拟面试三条关键链路的稳定性。

**2 分钟展开**

我会先承认项目阶段：这就是一个 workflow-first 的 MVP，不是成熟平台。它的长处是链路打通了，数据模型和状态管理也有基础；短板是核心耗时流程还跑在应用进程里，AI 调用虽然做了 timeout、retry 和 fallback，但还没形成完整的生产级治理。

为什么我优先重构任务体系，而不是先改 UI 或加更多功能？因为当前最贵的链路都依赖后台任务：简历解析、定制简历生成、面试问题池准备。如果任务可靠性不够，前面做再多产品功能都会被拖垮。

当前项目真实已实现：

- 有明确 workflow、有状态、有版本、有错误分类。
- 有基础可运行的后台任务和 AI 兜底策略。

当前项目没有实现，但如果生产化我会怎么优化：

- 第一优先级：独立任务队列和 worker。
- 第二优先级：AI 调用链 observability、熔断和成本控制。
- 第三优先级：更强的权限、审计和多租户边界。

**继续追问**

- 如果继续问“为什么不是先上缓存”，我会说当前瓶颈更可能在任务可靠性和 AI 调用，而不是简单读性能。
- 如果继续问“为什么不是先拆微服务”，我会说当前复杂度还没到必须拆服务的程度。

**避坑提醒**

- 不要说“项目已经能上线，只差一点优化”，这会显得缺乏判断力。
- 更好的回答是承认 MVP 边界，同时给出明确优先级。

### 6.3 线上如果出现 AI 超时、空响应、解析失败、存储失败、跨用户访问、状态不同步，你怎么排查？

**问题**

如果线上出现 AI 超时、空响应、解析失败、存储失败、跨用户访问、状态不同步，你怎么排查？

**30 秒短答**

我会先按链路分层定位：入口层看请求和用户身份，业务层看状态流转和版本，AI 层看 provider 错误分类和重试记录，存储层看 PostgreSQL / Redis / MinIO 是否正常。我的目标不是立刻猜结论，而是尽快定位故障发生在“鉴权、任务、AI、存储、数据一致性”中的哪一层。

**2 分钟展开**

我会举几个具体例子。

如果是 AI 超时或空响应，我会先看该请求的错误分类、provider、attempt 次数和 timeout 配置，确认是上游慢、连接断了，还是返回体为空。

如果是解析失败，我会区分是 PDF 本身问题、Markdown 清洗失败，还是结构化解析失败。resume 这条链路里我会重点看 `parse_status`、`parse_error`、`parse_artifacts_json` 和 parse job 状态。

如果是存储失败，我会先区分数据库写失败还是 MinIO 失败，因为两者的恢复方式不一样。数据库失败更偏事务和连接问题，MinIO 失败更偏对象存储依赖和网络问题。

如果是跨用户访问，我会优先查是不是所有权校验漏了某条查询。

如果是状态不同步，我会重点查任务是否中断、版本是否 stale、上游是否更新后没有触发下游失效标记。

当前项目真实已实现：

- 错误分类、日志、状态字段和版本字段已经提供了基础排查抓手。
- cross-user 测试已经覆盖部分数据隔离风险。

当前项目没有实现，但如果生产化我会怎么优化：

- 我会补 request id 贯穿日志和任务日志。
- 我会补结构化监控面板和报警。
- 我会补更系统的状态事件表和故障回放能力。

**继续追问**

- 如果继续问“你先看日志还是先看数据库”，我会说先按故障类型判断，但通常先确认用户影响面和哪个层最可疑。
- 如果继续问“你怎么判断是不是 AI 上游问题”，我会看 provider error、timeout、5xx、stream disconnect 等分类。

**避坑提醒**

- 不要回答成一堆 Linux 命令清单，先讲排查框架。
- 不要假装现在已经有完整 tracing，当前没有。

### 6.4 如果让你继续优化性能和稳定性，你的优先级是什么？

**问题**

如果让你继续做性能和稳定性优化，你的优先级会怎么排？

**30 秒短答**

我的优先级是：先保住主链路可用，再谈性能极致。第一优先级是后台任务可靠性，第二优先级是 AI 调用链治理，第三优先级是数据库和存储的可观测性，最后才是更细的缓存和吞吐优化。

**2 分钟展开**

我不会一上来就说“先做缓存”。因为这个项目当前最贵、也最脆弱的部分是异步任务和 AI 依赖。如果这些部分不稳定，单纯优化查询速度对用户感知帮助有限。

所以我会这样排：

第一，任务系统可靠性。因为 resume parse、tailored resume、mock interview prep 都靠它。

第二，AI 调用链治理。包括 provider 健康度、超时预算、重试上限、fallback 成功率和成本监控。

第三，数据库和对象存储的观测与优化。比如热点查询、索引质量、对象下载失败率、慢请求分析。

第四，必要时引入缓存和限流，但前提是真实流量证明它需要。

当前项目真实已实现：

- 已经有基础状态和错误分类，说明系统有一定可观测基础。
- 还没有做成重缓存、高吞吐优化型架构。

当前项目没有实现，但如果生产化我会怎么优化：

- 任务系统升级。
- AI 质量和稳定性仪表盘。
- 数据库慢查询治理和对象存储可靠性监控。
- 最后再做更精细的缓存和流量保护。

**继续追问**

- 如果继续问“为什么不先做数据库优化”，我会说当前主痛点更偏长链路和外部依赖。
- 如果继续问“缓存能解决什么”，我会说它更适合解决高频稳定读取，不适合替代 workflow 主状态。

**避坑提醒**

- 不要机械背“加缓存、加队列、加集群”，要贴合当前项目真实瓶颈。
- 不要把性能和稳定性混成一个词，要先判断是吞吐问题还是可靠性问题。
