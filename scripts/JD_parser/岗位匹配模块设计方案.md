# 《岗位匹配模块》重设计方案 V2（含外部调研）

## Summary
- 当前仓库已有岗位匹配雏形，但仍停留在“JD 结构化 + 单次匹配报告”阶段。现状证据在 [job.py](/Users/zhengwenze/Desktop/codex/career-pilot/apps/api/app/services/job.py)、[match_report.py](/Users/zhengwenze/Desktop/codex/career-pilot/apps/api/app/services/match_report.py)、[jobs/page.tsx](/Users/zhengwenze/Desktop/codex/career-pilot/apps/web/src/app/(dashboard)/dashboard/jobs/page.tsx)。
- 本次重设计把岗位匹配定义为《职点迷津》的“闭环中枢”，负责管理目标岗位、判断当前简历适配度、生成可执行改简历任务、输出模拟面试训练蓝图，并接收后续训练反馈回流。
- 已确认的产品取舍：
  - 模块定位：闭环中枢
  - 模拟面试联动：深度联动
  - 岗位来源：本期先手动录入为主，不做岗位发现/抓取

## External Benchmarks
- [Teal Job Matcher](https://help.tealhq.com/en/articles/9524864-using-the-job-matcher) 和 [Teal Auto-Select](https://help.tealhq.com/en/articles/9923251-using-job-matching-resume-curation) 的关键借鉴点是：岗位匹配不止展示分数，还要反向驱动“为该岗位裁剪简历内容”。
- [Teal Job Matcher Update - May 2025](https://help.tealhq.com/en/articles/11379501-job-matcher-update-may-2025) 说明成熟产品会把岗位匹配和岗位管理打通，而不是孤立成一个工具页。
- [Jobscan 官方说明（2026-01-20 更新）](https://support.jobscan.co/hc/en-us/articles/42869628183699-What-exactly-is-being-checked-Can-you-rate-my-resume) 强调匹配判定核心是硬技能、学历、职称、软技能和关键词，适合作为本项目规则层的维度基线。
- [Resume-Matcher GitHub](https://github.com/srbhr/Resume-Matcher) 的借鉴点是“master resume -> job-specific tailoring”的工作流，而不是只做 ATS 分数。
- 结论：岗位匹配模块应输出三种可复用资产，而不是一个数字：
  - `岗位画像`
  - `简历定制任务包`
  - `模拟面试训练蓝图`

## Implementation Changes
- 数据模型升级：
  - 保留 `job_descriptions` 作为目标岗位主表，但扩展为“岗位画像中心”，新增 `priority`、`status_stage`、`recommended_resume_id`、`latest_match_report_id`、`parse_confidence`、`competency_graph_json`。
  - 新增 `job_parse_jobs`，对齐简历解析的异步任务模型，状态统一为 `pending -> processing -> success|failed`。
  - 保留 `match_reports`，但改为“不可变匹配快照”，每次生成都绑定 `resume_id + resume_version + job_version`，新增 `fit_band`、`stale_status`、`tailoring_plan_json`、`interview_blueprint_json`。
  - 新增 `job_readiness_events`，记录岗位从“待分析”到“可投递”的状态变化，用于后续概览页和提醒。
- 领域对象重构：
  - `JobStructuredData` 从“技能/关键词/职责摘要”升级为“岗位画像”，包含 `must_have`、`nice_to_have`、`responsibility_clusters`、`experience_constraints`、`domain_context`。
  - `MatchReport` 输出从“strength/gap/actions”扩展为四层：
    - `scorecard`
    - `evidence_map`
    - `gap_taxonomy`
    - `action_pack`
  - `action_pack` 固定拆成：
    - `resume_tailoring_tasks`
    - `interview_focus_areas`
    - `missing_user_inputs`
- 匹配引擎重构：
  - 第一层 `规则对齐`：技能、关键词、学历、年限、城市、职责语义簇。
  - 第二层 `证据映射`：把 JD 要求映射到简历具体字段和文本证据，避免“有分无证据”。
  - 第三层 `AI 语义补偿`：只负责补充弱召回项、重写建议文案、生成面试题方向，不能成为唯一评分源。
  - 第四层 `行动生成`：把差距转成用户能执行的动作，而不是泛泛建议。
- 页面结构重构：
  - 左栏：目标岗位列表，展示 `岗位名 / 公司 / readiness stage / 最新 fit band / stale 标记`。
  - 中栏：岗位驾驶舱，展示 `JD 原文 / 结构化岗位画像 / 推荐简历 / 核心差距 / 历史快照趋势`。
  - 右栏：下一步动作区，只放三类 CTA：
    - `更新匹配`
    - `生成该岗位的简历定制方案`
    - `开始该岗位的模拟面试`
- 模块联动设计：
  - 与简历解析模块联动：
    - 简历解析成功后生成标准化“候选人证据画像”
    - 匹配时读取该证据画像，而不是直接扫散乱 `structured_json`
    - 简历手动校正或新版本保存后，关联岗位快照标记为 `stale`
  - 与简历优化模块联动：
    - 岗位匹配直接输出 `TailoringPlan`
    - 简历优化模块只负责把 `TailoringPlan` 转成编辑建议和改写草稿
    - 改完简历后回写岗位模块，刷新适配度趋势
  - 与模拟面试模块联动：
    - 岗位匹配直接输出 `InterviewBlueprint`
    - 包含 `题目主题`、`追问规则`、`能力维度 rubric`、`优先训练短板`
    - 面试结束后回写 `weak_competencies` 和 `follow_up_needed` 到岗位模块，更新 `status_stage`
- 状态机设计：
  - 岗位解析：`pending -> processing -> success|failed`
  - 匹配快照：`pending -> processing -> success|failed`
  - 岗位准备度：`draft -> analyzed -> matched -> tailoring_needed -> interview_ready -> training_in_progress -> ready_to_apply`
- 日志与排障：
  - 关键日志字段统一带 `request_id`、`user_id`、`job_id`、`resume_id`、`report_id`、`status_stage`
  - 岗位解析和匹配都要支持“卡住超过 120 秒”的排查路径，复用 AGENTS.md 里的状态流转方法

## Public Interfaces
- `POST /jobs`
  - 创建目标岗位，立即返回岗位记录和最新 `job_parse_job`
- `GET /jobs`
  - 支持 `keyword`、`parse_status`、`status_stage`、`stale`、`priority` 过滤
- `GET /jobs/{job_id}`
  - 返回岗位驾驶舱聚合视图，不再只是 CRUD 明细
- `POST /jobs/{job_id}/parse`
  - 重跑岗位结构化
- `POST /jobs/{job_id}/match-reports`
  - 入参 `resume_id`
  - 产出异步匹配快照
- `GET /jobs/{job_id}/match-reports`
  - 返回该岗位所有历史快照，含 `fit_band`、`stale_status`
- `GET /match-reports/{report_id}`
  - 返回完整 `scorecard + evidence_map + tailoring_plan + interview_blueprint`
- 下游约定：
  - 简历优化模块消费 `tailoring_plan_json`
  - 模拟面试模块消费 `interview_blueprint_json`
  - 两者都回写 `job_readiness_events`

## Delivery Phases
- Phase 1：把岗位解析从同步规则调用改成异步任务化，先统一状态流转、日志和轮询模型。
- Phase 2：重构匹配结果结构，先产出 `scorecard + evidence_map + gap_taxonomy`，不急着做复杂 UI。
- Phase 3：增加 `tailoring_plan_json`，让岗位匹配可以直接驱动简历优化模块。
- Phase 4：增加 `interview_blueprint_json`，让模拟面试模块可以直接按岗位短板开练。
- Phase 5：做岗位准备度和历史趋势，让岗位模块真正成为闭环驾驶舱。

## Test Plan
- 后端测试：
  - JD 解析任务状态流转
  - 匹配任务状态流转
  - AI 关闭或失败时规则结果仍可用
  - 简历新版本产生后旧报告 `stale`
  - `tailoring_plan_json` 和 `interview_blueprint_json` 字段完整且结构稳定
- 前端手工验证：
  - 创建 JD 后可看到解析进度和结构化结果
  - 选择简历生成匹配后可看到证据、差距、行动包
  - 修改简历后岗位页出现“报告已过期，建议重跑”
  - 从岗位页跳到模拟面试时能携带 `match_report_id`
  - 成功/失败后轮询停止正确
- 验收标准：
  - 用户在岗位页能明确知道“适不适合、差在哪里、先改什么、先练什么、现在处于求职准备的哪一步”。

## Assumptions
- 不修改历史 migration，只通过新增 migration 演进结构。
- 本期不做岗位抓取、职位推荐、投递追踪增强。
- 继续沿用现有 Apple 风格 UI 基线，不做复杂运营式仪表盘。
- AI 输出默认视为“建议层”，证据层和状态层必须可追踪、可回退、可测试。
