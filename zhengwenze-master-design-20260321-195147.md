## Problem Statement
你要做的不是“把现有优化会话再润色一下”，而是把 Career Pilot 的《专属简历模块》定义成一个独立、可信、可交付的结果物。

这个结果物的输入是：
- 用户已有主简历的结构化 JSON
- 用户已有主简历的完整 Markdown
- 目标岗位描述
- 目标岗位关键词
- 优化强度

这个结果物的输出必须是：
- 一份严格遵守原文保真规则的岗位定制简历 JSON
- 一份完整可投递的 Markdown 简历
- 一份能解释“哪些关键词被纳入、哪些不能纳入、为什么”的审计信息

核心不是“让模型写得更像 JD”，而是“在事实不变的前提下，让简历更贴近目标岗位且更适合 ATS 与招聘者阅读”。

## What Makes This Cool
- 用户看到的不再只是 `summary/work/projects` 三块草案，而是一份完整的“岗位定制版简历成品”。
- 系统不需要在“真实性”与“岗位贴合度”之间二选一，而是把二者都变成显式约束。
- 优化结果不仅能下载 Markdown，还能留下 `matchSummary + audit`，让后续岗位匹配、人工复核、用户编辑都有依据。

## Related Prior Designs
- `Career Pilot《职点迷津》四模块系统设计` (2026-03-20): 已明确“简历优化”应建立在统一事实底座之上，LLM 只做改写与评审。
- `Career Pilot 工程蓝图（Schema + JSON 契约 + 状态流）` (2026-03-20): 已明确 `resume.structured_json` 是 canonical facts layer，不应混入展示态或中间产物。

## Constraints
- 现有链路已经存在：`/tailored-resumes/workflows -> match_report -> resume_optimization_session`，不能假装它不存在。
- 现有优化 AI prompt 是 rewrite-only，且只允许改 `summary / work bullet / project bullet`，这与“完整简历 JSON 成品”目标不等价。
- 当前 `apply_resume_optimization_session()` 会把 `optimized_resume_json` 回写到 `resume.structured_json`。这意味着优化结果如果直接使用你给出的新 JSON 契约，会污染 canonical resume schema。
- 当前前端 `/dashboard/optimizer` 只支持 `summary / work_experience / projects` 三个 draft section，不支持教育、技能、证书、语言、自定义区块的完整成品编辑。
- 你的 prompt 明确禁止编造、禁止改时间线、禁止删掉重要事实、禁止硬塞关键词。这些不是“提示词建议”，而是产品约束。

## Premises
1. 《简历优化功能》的主输出对象必须是一个独立的 `tailored resume artifact`，不能直接等同于当前 `ResumeStructuredData`。
2. `resume.structured_json` 继续承担“用户主简历 canonical facts layer”；岗位定制结果是派生结果物，不是新的事实源。
3. `match_report` 继续承担“岗位理解、差距诊断、改写任务生成”的上游职责；但最终优化简历必须有自己的结果 schema、审计字段和 markdown。
4. 你给出的 prompt 应该作用在“完整成品生成”这一步，而不是塞进当前只改 bullet 的 rewrite-only prompt 中勉强复用。

## Approaches Considered
### Approach A: 在现有 `resume_optimization_session` 上继续打补丁
Summary: 复用当前 `optimized_resume_json`，往里加几个字段，前端继续沿用三块 draft section，再额外拼出你要的 JSON。
Effort: S
Risk: High
Pros:
- 改动面最小，能最快看到接口返回内容变化。
- 后端已有工作流、测试和下载逻辑可继续复用。
Cons:
- 这是明显的补丁方案，违反当前任务边界。
- `resume.structured_json` 和“岗位定制结果物”会继续混用，事实层会被污染。
- 前端仍然只能编辑三块内容，无法支撑完整简历成品。
Reuses:
- `apps/backend/app/services/tailored_resume.py`
- `apps/backend/app/services/resume_optimizer.py`
- `apps/frontend/src/app/(dashboard)/dashboard/optimizer/page.tsx`

### Approach B: 把“岗位定制简历”定义为独立结果契约，并让现有 workflow 生产它
Summary: 保留现有 `tailored-resumes -> match_report -> optimization_session` 工作流，但把最终结果重定义为独立的 `TailoredResumeDocument`，其中包含你要求的 JSON 全量字段、完整 markdown、以及 audit；主简历 canonical schema 保持不变。
Effort: M
Risk: Low
Pros:
- 路径短，而且逻辑正确，符合“最短路径但不打补丁”。
- 保住现有工作流复用价值，同时把“主简历事实层”和“岗位定制产物”彻底分开。
- 你的 prompt 可以原样服务于最终产物生成，不必被压缩成只改 bullet 的 prompt。
Cons:
- 需要收敛一套新的结果 schema，并调整 session 持久化字段。
- 现有 optimizer 页面要改为“完整成品预览/局部编辑”，不是仅三块草案。
Reuses:
- `apps/backend/app/services/tailored_resume.py`
- `apps/backend/app/services/resume_optimizer.py`
- `apps/backend/app/schemas/tailored_resume.py`
- `apps/frontend/src/app/(dashboard)/dashboard/resume/page.tsx`

### Approach C: 两阶段生成，LLM 只产“候选改写”，后端再确定性组装完整 JSON
Summary: 上游仍由 `match_report` 生成 rewrite tasks；LLM 只返回各 section 的候选文字片段和关键词命中说明；最终完整 JSON 和 markdown 由后端按主简历结构确定性装配。
Effort: L
Risk: Medium
Pros:
- 最强可控性，最容易做 fact-preserving 校验。
- markdown 与 JSON 一致性最好，不容易出现字段与成品脱节。
- 以后接不同模型时，稳定性更高。
Cons:
- 实现成本高于当前阶段所需。
- 会把这次需求往“自建简历装配器”方向拉，超出最短路径。
Reuses:
- `apps/backend/app/schemas/resume.py`
- `apps/backend/app/services/resume_optimizer.py`
- 现有 fact check 逻辑与 markdown renderer

## Recommended Approach
选择 Approach B。

原因很简单：它是当前仓库里唯一同时满足三件事的路径：
- 不打补丁
- 不污染主简历事实层
- 能最短路径承接你已经写好的完整 prompt

Approach A 的问题不是“不够优雅”，而是逻辑上错了。当前 `resume.structured_json` 的职责是“用户主简历事实底座”；你要的新 JSON 结果包含 `matchSummary`、`audit`、`missingButImportantKeywords`、`markdown` 这些岗位定制派生信息，不能再被当成同一种对象。

Approach C 长期很强，但对这次需求来说过重。

## Detailed Design
### 1. 产品对象重新定义
新增一个独立结果对象，暂定名：

`TailoredResumeDocument`

它表示“基于某份主简历和某个目标岗位生成的一份专属简历成品”，不是主简历本体，也不是编辑中的临时 draft。

建议主结构直接对齐你给出的 JSON：

```json
{
  "matchSummary": {
    "targetRole": "",
    "optimizationLevel": "conservative",
    "matchedKeywords": [],
    "missingButImportantKeywords": [],
    "overallStrategy": ""
  },
  "basic": {
    "name": "",
    "title": "",
    "email": "",
    "phone": "",
    "location": "",
    "links": []
  },
  "summary": "",
  "education": [],
  "experience": [],
  "projects": [],
  "skills": [],
  "certificates": [],
  "languages": [],
  "awards": [],
  "customSections": [],
  "markdown": "",
  "audit": {
    "truthfulnessStatus": "passed",
    "warnings": [],
    "changedSections": [],
    "addedKeywordsOnlyFromEvidence": true
  }
}
```

### 2. 与现有 canonical resume 的关系
保留：
- `resume.structured_json`: 用户主简历事实层
- `resume.parse_artifacts_json.canonical_resume_md`: 主简历规范化 Markdown

新增原则：
- `TailoredResumeDocument` 是派生结果，不回写到 `resume.structured_json`
- 用户“应用到当前简历”时，系统只能把 `TailoredResumeDocument` 中与 canonical facts schema 对应的事实兼容字段映射回 `ResumeStructuredData`
- `matchSummary`、`audit`、`markdown` 这类岗位定制派生字段永远不进入 `resume.structured_json`

这一步是整个设计里最关键的边界。

### 3. 现有 workflow 如何落地新对象
保留现有主流程：
1. 选择主简历
2. 输入或更新目标岗位
3. 生成/刷新 `job`
4. 生成 `match_report`
5. 创建 `resume_optimization_session`
6. 生成 `TailoredResumeDocument`

其中第 6 步不再只产生：
- `optimized_resume_json`
- `optimized_resume_md`
- `draft_sections`

而是产生：
- `tailored_resume_json`: 完整 `TailoredResumeDocument`
- `tailored_resume_md`: 等于 `tailored_resume_json.markdown`
- `editor_sections_json`: 从完整成品中投影出的可编辑 section 视图
- `audit_report_json`: 与 `tailored_resume_json.audit` 对齐的可机读审计结果

### 4. Prompt 的正确放置方式
你给出的 prompt 不应该替换当前 `rewrite_only.txt`。

原因：
- 当前 `rewrite_only.txt` 明确只允许改 `summary/work/project bullet`
- 你给出的 prompt 要求输出完整 JSON 成品
- 两者输入边界、输出边界、产品目标都不同

正确做法：
- 保留 `rewrite_only.txt` 作为局部改写器，服务于低成本 fallback 或局部刷新
- 新增一个 `tailored_resume/full_document.txt`，直接使用你这版 prompt 作为完整成品生成器

这个完整成品生成器的输入应是：
- `job_description`
- `job_keywords`
- `original_resume_json`
- `original_resume_markdown`
- `optimization_level`
- 可选：`match_report_snapshot` 作为辅助证据，而不是事实源

### 5. 为什么 `match_report` 仍然有价值
虽然你的 prompt 只显式依赖原简历与 JD，但现有 `match_report` 不应该被绕开。

它仍然承担三类上游价值：
- 帮系统先提炼 JD 关键词与职责重点
- 先标出差距项与需谨慎处理的 missing keywords
- 为前端提供“为什么这么改”的任务上下文

但要注意：
- `match_report` 只能辅助改写策略
- 不能成为新增事实的依据

### 6. 持久化建议
不要推翻现有 session 表，但需要让它真正承载“专属简历成品”而不是仅承载三块 draft。

建议在 `resume_optimization_sessions` 中以 JSON 列为主保存：
- `tailored_resume_json`
- `tailored_resume_md`
- `editor_sections_json`
- `audit_report_json`

现有这些字段可以保留但职责要收窄：
- `optimized_resume_json`: 仅用于“可回写到主简历 facts layer 的兼容事实投影”
- `optimized_resume_md`: 可废弃或改为镜像 `tailored_resume_md`
- `draft_sections_json`: 被 `editor_sections_json` 替代
- `fact_check_report_json`: 升级为 `audit_report_json` 的一部分

### 7. 前端正确交互
当前 `/dashboard/resume` 已经是“专属简历模块”的入口，这是对的。

但当前展示只有：
- 主简历 markdown
- JD 输入
- 生成后的 markdown 成品

而你这次真正需要的是：
1. 生成后直接展示完整 `TailoredResumeDocument`
2. 让用户看见 `matchedKeywords` 与 `missingButImportantKeywords`
3. 让用户看见 `audit.warnings`
4. 提供按 section 编辑能力，但编辑对象来自完整成品，不是三块字符串
5. 下载的 markdown 必须等于 `tailored_resume_json.markdown`

因此前端应该从“优化草案编辑器”转向“专属简历成品编辑器”。

### 8. 真实可信校验
现有 `resume_optimizer.py` 已经有一部分 fact check 逻辑，这一层应该继续保留，并升级为对你新 JSON 契约的审计器。

必须检查：
- 是否新增公司、学校、项目、技能
- 是否改动 title/date/time range
- 是否新增数字、比例、成果强度措辞
- `matchedKeywords` 是否都能在原简历证据中找到
- `missingButImportantKeywords` 是否确实来自 JD 且原文无证据
- `markdown` 是否完整覆盖主要 section，而不是只剩摘要

`audit.truthfulnessStatus` 的来源不能只靠模型自评，必须由服务端校验结果决定。

### 9. “应用到当前简历”的正确语义
当前按钮语义是：
- 把优化结果写回 `resume.structured_json`
- 简历版本号 +1
- 旧 match report 标记 stale

这个方向是对的，但新的实现必须变成：
- 从 `TailoredResumeDocument` 中提取 canonical-compatible fields
- 映射回 `ResumeStructuredData`
- 只写回事实兼容层，不写回 `matchSummary / audit / markdown`

否则一旦写回完整定制结果对象，主简历就不再是“主简历”了，而变成“某次岗位定制视图”。

## Existing Code Reuse Map
- `apps/backend/app/services/tailored_resume.py`
  继续负责专属简历 workflow 编排，不必重写入口。
- `apps/backend/app/services/resume_optimizer.py`
  继续复用 fact check、markdown renderer、session 状态管理，但要把“结果对象”从三块 draft 升级为完整成品。
- `apps/backend/app/prompts/tailored_resume/rewrite_only.txt`
  保留为局部改写器，不承担完整成品职责。
- `apps/backend/app/schemas/tailored_resume.py`
  应成为完整成品 schema 与接口 contract 的主入口。
- `apps/frontend/src/app/(dashboard)/dashboard/resume/page.tsx`
  继续作为《专属简历模块》主页面，承担生成、预览、下载、进入编辑。
- `apps/frontend/src/app/(dashboard)/dashboard/optimizer/page.tsx`
  若保留，应改造成完整成品编辑页；若不保留，则把编辑能力回收进专属简历主页面。

## Open Questions
- `customSections` 是否需要支持任意标题自由编辑，还是先只做只读保留。
- `basic.title` 是否来自原简历已有标题，还是允许基于目标岗位做保守改写。
- `links` 的结构是字符串数组即可，还是需要对象化 `{label, url}`。
- `optimization_level` 是否只先支持 `conservative`，还是预留 `balanced/strong`。

## Success Criteria
- 用户上传一份主简历并输入 JD 后，能得到一份完整 `TailoredResumeDocument`。
- 返回结果严格是一个 JSON 对象，并包含完整 markdown 简历。
- markdown 覆盖主简历主要 section，不退化成摘要。
- 审计层能明确给出 `matchedKeywords`、`missingButImportantKeywords`、`warnings`。
- “应用到当前简历”后，`resume.structured_json` 仍保持 canonical facts schema，不被岗位定制字段污染。
- 前端可以完整预览专属简历成品，并下载与页面一致的 markdown。

## Next Steps
1. 先在后端冻结产品对象：定义 `TailoredResumeDocument` schema 和 session 持久化边界。
2. 新增完整成品 prompt 文件，把你给出的 prompt 原样落成模板，不去硬塞进 `rewrite_only`。
3. 改造 workflow 生成逻辑，让 `tailored_resume.py` 产出完整 document，而不是只产出三块草案。
4. 再改前端预览与编辑层，把《专属简历》页面升级为完整成品视图。
5. 最后才处理“应用到当前简历”的映射回写，确保事实层边界不被破坏。

## What I Noticed About How You Think
- 你没有说“做个简历改写功能”，你说的是“原文保真 + 岗位定制优化”，这说明你在意的不是模型文风，而是产品可信度。
- 你把“真实可信”“尽量保留原文信息”“只输出一个合法 JSON 对象”写成硬约束，这不是提示词堆砌，而是在先定义产品边界。
- 你明确写了“禁止为了匹配 JD 而硬塞不存在的关键词”，这说明你已经抓住了这个模块最容易失真的地方。
- 你给出的 JSON 不是随手列字段，而是把 `matchSummary / markdown / audit` 都纳入了结果对象，说明你想要的是完整交付物，不是中间变量。
